# 第三步：存储文件、处理资料并生成输入源

## 目标

在用户完成第二步确认后，再把确认纳入的输入统一沉淀到工作目录，供后续需求归纳和测试用例生成使用。

## 原则

- 没有用户明确确认，不进入第三步。
- 未经用户确认，不提前把可疑文件纳入正式输入。
- 所有整理后的资料尽量统一为 Markdown。
- 保留来源追溯信息，不要丢失原始输入上下文。
- 不额外创建未明确提及的中间文件。

## 处理规则

### Google Docs

- 优先读取公开内容。
- 能访问时整理为 `source-google-doc.md` 或带序号的同类文件。
- 不能访问时在 `open-questions.md` 中记录阻塞。

### 对话上传文件

- 优先把可读内容转成 Markdown。
- 如果本身就是 Markdown，可直接清洗整理。
- 如果是其他文本类文件，提炼关键信息后转写为 Markdown。

### Figma

- 提炼页面、关键流程、组件状态、字段规则、校验、文案、提示、按钮行为和跳转关系。
- 不把未标注交互写成确定事实。
- 产出为 `source-figma.md` 或带序号的同类文件。

### 本地文件或目录

- 只处理第二步确认纳入的文件。
- 本地 Markdown 文件可直接复制或清洗后写入工作目录。
- 其他文本类文件先提炼关键信息，再转写为 Markdown。
- 非文本类文件如果暂时无法稳定解析，保留路径并在 `open-questions.md` 中说明限制。

## 文件创建约束

- 只在真正需要时创建资料文件和最终产物。
- 如果当前任务只需要 `input-manifest.json` 和最终测试用例产物，就不要补其他辅助文件。
