# Everoute 分布式防火墙 UI Testcase Rules

## 必须遵守

- Step6 先读取 `everoute_dfw_product_knowledge.md`，用其补齐 DFW 的对象树、策略类型、对象来源、日志/计数/许可/联动知识，再回到 common 骨架组织结构。
- 当前 DFW 历史 testcase 样本不足，首版规则以 `everoute_common_testcase_rules.md` 为主，本文件仅作为补充规则使用。
- 如果当前需求显式涉及 DFW 关联集群、策略对象、策略列表、规则创建编辑、权限或文案调整，必须保留 DFW 专属分组，不允许完全回退到 common 规则。
- 与集群关联、策略对象、策略规则、文案变化、权限限制相关的场景，应优先独立分组，不要混进“其他改动”。

## 常见分层模式

- DFW 业务对象 -> 列表 / 创建 / 编辑 / 删除 / 详情 -> 字段或校验点 -> `[C]`
- DFW 业务对象 -> 关联集群 -> 选择条件 / 过滤逻辑 / 报错文案 -> `[C]`
- DFW 业务对象 -> 权限管理 / 异常限制 / 事件审计 -> 动作类型 -> `[C]`

## 展开规则

- 高优先级展开轴：策略类型、策略对象来源、白名单/黑名单来源、对称策略、日志/监控模式、关联集群状态、许可类型、文案变化、权限、异常限制。
- 如果 DFW 场景与 LB 或 VPC 有联动，优先保留“联动关系”这一层，再展开容量状态和结果。
- 如果当前需求只包含 DFW 的文案或联动变化，不必强行扩展完整 DFW 产品矩阵，只保留与本次改动直接相关的分组和组合。
- 没有历史样本时，Step7/8 仍必须按产品知识摘要中的对象树、策略类型和对象来源继续展开，不能因为缺样本而退化成粗粒度 common 分组。

## 测试资源类型

### DFW 对象资源

- 关联集群
- 策略对象
- 策略规则
- 权限角色
- 审计动作

### DFW 许可资源

- 许可对象：关联集群、策略对象
- 许可方式：按关联集群的物理 CPU 插槽、按策略对象 / VM
- 额度状态：充足 / 刚好 / 不足 / 已满
- 结果资源：报错只包含 DFW、DFW 文案变化、DFW 限制结果

### DFW 联动资源

- 关联集群状态
- 与 LB 的联动
- 与 VPC 的联动
- 文案变化

## 资源展开提示

- 只要场景涉及 DFW 许可，就必须显式保留 `许可对象 + 许可方式 + 关联集群状态 + 结果归属`。
- 只要场景涉及联动，必须显式保留“结果归属”这一层，避免把报错泛化掉。

## 禁止项

- 禁止在 DFW 样本不足时凭空扩展大量产品专属分层。
- 禁止把 DFW 相关改动完全泛化成 common 规则，导致“报错只包含 DFW”“DFW 文案变化”等专属结果消失。
- 禁止把 DFW 与其他产品联动结果压缩为单一“报错变化”分组。
- 禁止把 `放行/拒绝`、`虚拟机/IP/Pod`、`日志/计数/查询` 混成单条 case。
