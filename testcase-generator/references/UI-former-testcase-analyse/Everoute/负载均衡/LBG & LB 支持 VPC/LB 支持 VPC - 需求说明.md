# 概述

## **背景**

目前 Everoute LB 可以为 Underlay 网络中的客户端及后端服务器提供四层负载均衡服务。Eveorute  支持 VPC 后，VPC 中的业务也需要 LB 功能作为支撑。

VPC 的一个关键使用场景是跨集群高可用，即通过 VPC 将多个集群的业务网络打通，形成一个大二层网络空间，在其中运行多个业务虚拟机，当其中一个集群或数据中心故障时，可以由其他虚拟机继续提供服务，保证业务不间断，在此场景下，LB 对日常流量的负载均衡以及故障场景下对后端服务器的健康检查能力就变得更加重要。

相关需求：[\#46](http://gitlab.smartx.com/feedback/everoute/-/issues/46) 

## **用户故事**

* 作为基础设施管理员，我对公司的业务网络进行了改造，将原本的 VLAN 网络改造为 VPC 网络，更好地满足部门间的环境隔离需求。某部门存在多实例的业务需求，客户端在集群外部，后端服务器在集群中作为虚拟机运行。前期在 VLAN 环境下使用 LB 进行负载均衡， 业务迁移到 VPC 后，仍希望继续使用 LB，对外部提供服务。  
* 作为基础设施管理员，我规划了两个数据中心作为主备，提供业务高可用方案，两数据中心各部署一个物理集群，集群间的业务网络三层互通。我希望借助 Everoute 网络能力可以达到如下效果  
  * 将多实例业务的多个虚拟机分别部署在两个数据中心，对单个业务来讲，选择其中的一个数据中心作为主站点，另一个数据中心作为备用站点。  
  * 两站点中的多个虚拟机在同一子网内，网络相互连通。  
  * 非故障场景下仅由主站点的虚拟机对外提供服务。  
  * 主站点故障时，Everoute 可以自动检测到故障，切换为由备站点的虚拟机提供服务。此时保持对外提供服务的 IP 地址不变，或自动宣告新的路由规则，告知客户端由新的路由继续访问此服务。

## **支持方式**

通过 Everoute LB 为 VPC 内的业务提供负载均衡能力，支持如下能力。

* 复用已有的 LB 功能及 UI 页面，为 VPC 内的客户端或后端服务器提供同等能力（**不支持** DNAT 模式）。  
* 对于每个虚拟服务，如果客户端或后端服务器位于 VPC 内，支持将 LB 中虚拟服务使用的 VIP 或 LIP 地址也配置在 VPC 内部，包含如下场景（此次新增场景二和场景三）：

## **运维管理**

### **负载均衡虚拟机**

如需在 VPC 中使用 LB 能力，需先为 LB 虚拟机启用支持 VPC 的能力，并为其配置 TEP IP 地址，用于与 VPC 内部进行隧道通信。

此需求仅针对每个 Everoute 服务中只有一组 LB 虚拟机的场景描述，支持多 LB 虚拟机的情况在单独的需求中说明。

#### 关闭功能

如需为 LB 虚拟机关闭「用于 VPC 网络」特性：

* 前提条件：LB 虚拟机中，“与客户端通信”的网络和“与服务器池通信”的网络，均未关联任何 VPC 子网。  
* 取消勾选后，释放已配置的 TEP IP 地址，下次勾选时需重新配置。

### **虚拟机网络关联**

#### 关联要求

在 VPC 中使用 LB 功能前，需先为 LB 关联与客户端通信及与后端服务器通信使用的 VPC 子网，子网需要与客户端/服务器池位于同一个 VPC 内。

关联后，将在此子网中分配如下 3 类 IP 地址：

* 内部 IP 地址  
  * 用于 vrrp 通信，对 VS 的主备实例进行健康检测。  
  * 每关联一个与客户端通信的 VPC 子网，需为每个 LB 虚拟机分配一个内部 IP。  
  * 如 VIP 在 VPC 外部，则默认采用 169.254.x.x 地址段分配内部 IP 地址。  
  * 如 VIP 在 VPC 内部，需从 VIP 所在的 VPC 子网内分配此 IP 地址。  
* 虚拟服务 VIP 地址  
  * 用于与客户端通信。  
  * 位于与客户端通信的 VPC 子网内。  
  * 创建虚拟服务时分配。  
* LIP 地址  
  * 用于与后端服务器通信。  
  * 每关联一个与服务器池通信的 VPC 子网，需为每个 LB 虚拟机分配一个 LIP。

用户可以根据 IP 分配要求选择：

1. 直接关联客户端/服务器池所在的子网，在此子网内分配 LB 所需的 IP 地址。

   如图，以东西向为例，可以直接关联客户端所在的 Subnet1/Subnet2 用于与客户端通信，在其中配置 vip 地址；关联后端服务器所在的 Subnet3/Subnet4 用于与服务器池通信，在其中配置 lip 地址。此方式下 LB 服务所用 IP 与客户端/服务器池所用 IP 共用一个子网的地址池。

| 直接关联客户端/服务器池所在的子网 |  |
| :---: | :---: |
| **南北向** | **东西向** |
| ![][image7] | ![][image8] |

2. 或，为 LB 服务单独创建一个子网，用于上述地址的分配和管理。

   如图，单独为 LB 服务创建一个 Subnet5，用于与此 VPC 内的客户端及服务器池通信，所有 LB 服务相关的 IP 地址均从此子网的地址池内划分，推荐使用此方式，使各用途下的地址分配更加清晰明确。

   进一步，也可以考虑单独创建两个子网 Subnet5 和 Subnet6，分别用于与客户端通信和与服务器池通信。

#### 关联与客户端通信的网络

关联网络前，需先选择「网络类型」：

* VLAN 网络  
* VPC 网络

选择关联 VLAN 网络时：

* 增加提示：关联网络后，将为每个负载均衡虚拟机自动配置一个内部 IP 地址，用于对虚拟服务中一组负载均衡虚拟机内的主备实例状态进行检查，内部 IP 地址从 169.254.0.0/16 网络中自动分配。  
* 其他配置内容与当前版本相同

选择关联 VPC 网络时，需配置如下内容：

* VPC  
  * 提示：需选择客户端所在的 VPC。  
  * 选择当前 Everoute 服务下已有 VPC。  
* VPC 子网  
  * 提示：用于分配内部 IP 地址及虚拟服务的 VIP 地址，可选择客户端所在的 VPC 子网，或为负载均衡服务单独创建子网。  
  * 选择上述 VPC 中已有子网，且尚未关联为与客户端通信的网络。不支持在此处创建一个新的 VPC 子网。  
* 内部 IP：为每个 LB 虚拟机分配一个内部 IP，用于主备实例健康状态检查  
  * 提示：用于对虚拟服务的主备实例状态进行检查，需选择 VPC 子网内的 IP 地址。  
  * 配置方式：手动指定 VPC 子网内任一可用的 IP 地址，或留空自动分配。  
  * 校验内容  
    * 子网内可用的 IP 地址数不小于 LB 虚拟机数。  
    * 手动指定的地址均包含在所选的 VPC 子网 CIDR 内，非网关地址，且未被其他虚拟机使用。

#### 关联与服务器池通信的网络

* VLAN 网络：与当前版本的关联步骤相同。  
* VPC 网络：关联方式如下。

* VPC 子网

  * 选择上述 VPC 中已有子网，且尚未关联为与服务器池通信的网络。不支持在此处创建一个新的 VPC 子网。

#### 编辑关联

上述 VPC 网络关联后，关联的 VPC、VPC 子网，内部 IP 地址以及 LIP 地址均不支持修改。

#### 取消关联

取消关联的限制不变，即

* 用于与客户端通信：如当前网络未在任何一个虚拟服务作为与客户端通信的网络，则允许取消关联，否则不允许。  
* 用于与服务器池通信：如当前网络未在任何一个虚拟服务作为与服务器池通信的网络，则允许取消关联，否则不允许。

取消关联后，释放对应的内部 IP 地址及 LIP 地址，如后续重新关联此网络，需重新设置。

## **服务器池**

### **支持 VPC**

支持 VPC 后，服务器池配置方式不变，

* 在成员信息中，可输入 VPC 子网内的 IP 地址。  
* 「网关」字段  
  * 由于 lip 和后端服务器在同一个 VPC 中，无需配置网关。  
  * 提示信息中增加：如服务器池成员在 VPC 子网内，无需填写网关。  
  * 由于无法区分成员地址是否也存在于 VLAN 网络中，因此此处仅做提示，不进行校验。

### **支持备成员**

已有版本中的成员输入方式存在以下优化项，在此需求中一并说明（此内容与支持 VPC 不直接相关）。

* 配置备用成员   
  * 在服务器池的所有成员中，允许将一个成员标识为「备用成员」。  
  * 提示，此成员仅在所有其他服务器均不可用时，进行流量处理。  
  * 工作方式如下（后端依赖 LVS 实现，其中会具体区分为两种模式）🟡2025-07-02 根据讨论补充  
    * 负载均衡模式  
      * 支持多个主成员 \+ 1 个备成员  
      * 预期流量均在主成员中调度，只有主成员全部异常时，才会 fallback 到备用成员，备成员通常提供的是 Error 界面的展示，起到告知作用， 而不正常提供业务响应  
      * 此时无法为备成员提供健康检查，所有主成员故障时，则流量调度到备成员  
    * 主备模式  
      * 支持 1 个主成员 \+ 1 个备成员  
      * 预期流量主/备成员中，根据健康检查情况调度，类似于 LBG 的中的切换方式  
      * 此时主备成员均可提供健康检查，主成员故障时，如此成员健康，则由备成员进行流量处理，主成员恢复时再自动切回使用主成员。  
  * 此版本中，界面未区分两种模式，只支持配置一个备用成员，后端根据实际启用的主成员数量判断  
    * 只有一个启用状态的主成员时，按“主备模式”生效，备成员支持健康检查  
    * 有超过一个启用状态的主成员时，按“负载均衡模式”生效，备成员无健康检查，因此备成员也无健康状态  
  * 支持备成员后，对服务器池整体健康状态的影响如下

| 健康检查 | 模式 | 主成员状态 | 备成员状态 | Pool 状态 |
| ----- | :---- | :---- | :---- | :---- |

  * 限制内容  
    * 服务器池中，只能有一个备用成员。  
    * 此成员不支持配置权重、启用状态。  
    * 服务器池除了备用成员外，还有至少一个启用状态的其他成员。（即只有一个启用状态的成员时，此成员不可选为备用）  
    * 只有一个主成员时，不支持配置权重。  
* 每添加一个新成员，「启用」字段均默认勾选（当前行为是默认不勾选）。

# 相关影响

## **网关服务**

🟡 2025-06-12 根据大家保险使用场景增加对网关服务的适配，[Slack 讨论](https://smartx1.slack.com/archives/C014SKH8823/p1749460657645589)

虚拟服务中，与客户端或服务器池通信的网络位于 VPC 子网的时候，其 VIP 地址或 LIP 地址会作为 VPC 内部 IP 提供

* 客户端/后端服务器位于 VPC 内时，可直接在 VPC 内部访问 VIP/LIP，无需额外配置；  
* 客户端/后端服务器位于 VPC 外时，可支持通过如下网关服务访问 VIP/LIP  
  * NAT 网关：支持为 VIP/LIP 配置 SNAT/DNAT 规则，允许与外部通信  
    * 现状：当前配置 DNAT 规则时，内部目标地址字段存在校验，必须填写已经分配给虚拟机的 IP 地址，否则限制不允许提交，由于当前 Tower 无法识别 IP 地址已被分配给 LB 虚拟服务，因此此处无法填写 VIP  
    * 当前调整：取消上述校验条件，不再检查 IP 地址是否已分配 [TOWER-18607](http://jira.smartx.com/browse/TOWER-18607)   
    * 后续调整：增加对 LB VIP 地址的识别，如目标地址未分配给虚拟机或 LB VIP，给出相关提示（[Slack 讨论](https://smartx1.slack.com/archives/C014SKH8823/p1749463351830129?thread_ts=1749460657.645589&cid=C014SKH8823)）  
  * 路由网关：支持 VIP/LIP 通过路由网关与外部 通信

    当前后端无法支持，需要在 namespace 里为不同 vpc 子网中的 vip 增加默认路由到子网网关 [ER-1212](http://jira.smartx.com/browse/ER-1212) 

* 对等连接：目前的实现可天然支持，无需调整，在此周期中测试验证即可。

  L2 网关、浮动 IP 在当前版本暂不支持。

## **VPC 子网**
