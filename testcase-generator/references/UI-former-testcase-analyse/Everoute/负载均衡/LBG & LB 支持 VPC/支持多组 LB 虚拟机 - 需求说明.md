# 相关概念

* LB 实例（Load Balancing Instance，LBI）：表示一组（2-4 个）用于部署 LB 服务的虚拟机。

   对应实现文档中的 LB Group、LB 集群。

  说明：Everoute 2.0.0 版本中，每个 Everoute 服务中只支持部署一组此类虚拟机，因此没有专门进行命名，均以「负载均衡虚拟机」代称。从 Everoute 3.4.0 版本开始，支持部署多组此类虚拟机，也支持在两组虚拟机之间建立主备关系，因此为方便理解、避免歧义，需要一个统一的概念名词。

* LB 实例组（Load Balancing Instance Group，LBG）：两个 LB 实例建立主备关系后，组成一个 LB 实例组，用于提供实例的高可用。

  对应实现文档中的 AE Group。

* 服务器池组（Server Pool Group）：将两个服务器池建立主备关系，组成一个服务器池组，当主 Pool 中的成员全部故障时，可自动切换至使用备 Pool 中的成员，提供后端服务器层面上的高可用。

# 概述

## **背景**

目前每个 Everoute 服务中，支持部署一组（2-4 个） LB 虚拟机，以提供负载均衡服务。如果用户部署了多个 SMTX 集群（如佳通轮胎），也只能使用同一组 LB 虚拟机，流量路径和负载分配都不灵活；只能部署多个 Everoute 服务后才能增加 LB 虚拟机，增加了管理开销。后续会通过 LB 支持 VPC 内的负载均衡，并扩展 Everoute 整体的高可用方案，同时也将在企业云中引入 LB 能力，支持多租户下的负载均衡。因此一组 LB 虚拟机从隔离性和流量负载上都难以继续满足要求。

在此背景下，需要增加支持多组负载均衡虚拟机，管理员可以按需将它们部署在同一 Eveorute 服务范围内的一个或多个 SMTX 集群上，并可以通过两组虚拟机组成主备，增强 LB 的高可用能力。

## **典型场景**

### **多 SMTX 集群/多租户**

用户可能根据不同的业务需求部署多个 SMTX 集群，希望多集群之间的业务流量各自隔离，并单独进行负载分配。此场景下希望为每个集群配置一组单独的 LB 虚拟机，每个业务集群根据自身的需要单独配置 LB 资源并独享 LB 虚拟机的所有负载均衡能力。

![][image1]

25 年将推出的企业云方案中，可以通过自服务平台（SSC）提供多租户能力，将企业内的各个分公司、部门划分为不同的租户，实现租户之间的资源和管理隔离。

网络层面，可以通过 VPC 实现租户之间的网络资源隔离，同时由于不同租户内均可能存在负载均衡需求，需要支持对租户间的负载均衡功能进行隔离管理，即可以为每个租户单独规划 LBI，保证租户间的网络安全，同时避免资源争抢。

![][image2]

### **跨集群灾备**

灾备场景下，可以采用如下部署方案：

* 部署两个 SMTX 集群，分布在不同位置的两个数据中心内，互为主备，提供**数据中心级别的高可用**。  
* 在 LB 实例层面上，可以部署两个 LBI，分别位于两个 SMTX 集群，组成一个 LBG。

后端服务器的部署方式，与业务类型、是否使用 VPC 网络、以及业务本身采用的灾备方式（多实例、备份、异步复制）有关，因此可能的场景组合较多，此处仅以一个典型场景为例进行介绍，更多灾备场景的说明详见[高可用场景分析](#场景说明)。

对于多实例的业务，可以在两个集群分别部署多个后端服务器，并分配到两个服务器池中，组成服务器池组。

非故障场景下，只通过主集群中的 LBI 和服务器池中提供服务，此时主备服务器池中的成员均为健康。

如主集群（或数据中心）整体故障，会通过控制面对 LBI 进行故障切换，将所有虚拟服务及所有 VIP 迁移至 LBI-B 中；并通过健康检查中对服务器池的主备进行切换，切换为由 Pool-B 提供服务。

## **目标**

1. 支持在一个 Everoute 内部署多个 LB 实例。  
2. 支持两个 LB 实例关联为主备关系，组成 LB 实例组，提供 LBI 级别的高可用。  
3. 支持配置将两个服务器池关联为主备关系，组成服务器池组，提供服务器池级别的高可用。

# 运维管理

## **负载均衡实例（LBI）**

### **基础设定**

* 每个 LB 实例均由 2-4 个 LB 虚拟机组成，一个实例内的所有 LB 虚拟机均需要位于同一个 SMTX 集群内（与 Everoute 2.0 中设定相同）。  
* 每个 Everoute 服务内，允许创建多个 LB 实例，多个实例可以位于相同 SMTX 集群或不同集群。

### **创建**

已有版本中，仅支持在部署 Everoute 服务和启用 LB 功能时，支持创建唯一的一个 LBI。

新版本中，需保持上述创建方式不变，在服务设置中，单独增加「创建负载均衡实例」操作。

需填写字段如下：

旧版本中只存在一个 LBI，不支持单独删除，此版本需增加删除 LBI 操作。

* 删除单个 LBI  
  * 前提条件：LBI 中没有任何虚拟服务。  
  * 删除操作：自动删除 LBI 中的所有负载均衡虚拟机。  
* 卸载负载均衡功能，或卸载 Everoute 服务  
  * 前提条件：Everoute 中没有任何 LBI 或 LBG。

### **创建**

可选择满足如下要求的两个 LBI，关联为主备关系，并创建 LB 实例组。

* 未关联到任何 LBG；  
* LBI 中未关联任何网络（包括与客户端通信和与服务器池通信的网络）；  
* 主备 LBI 中，如下配置均相同：  
  * LB 虚拟机数量  
  * 配置类型  
  * 是否支持 VPC 网络

创建时，需指定如下内容：

| 字段名 | 说明 | 是否必须 | 是否支持编辑 |
| ----- | ----- | :---: | :---: |
| 名称 | 填写 LB 实例组的名称，在一个 Everoute 服务内不允许重复。 | ✓ | ✓ |
| 描述 | 填写 LB 实例组的描述，非必须。 |  | ✓ |
| 主负载均衡实例 | 可选当前 Everoute 服务中，满足上述要求的 LBI。 选择时展示：名称、所属集群、负载均衡虚拟机数量、配置类型、是否用于 VPC 网络。 | ✓ |  |
| 备负载均衡实例 |  | ✓ |  |

创建 LB 实例组后，如主备 LBI 属于同一集群，需自动执行如下操作：

* 为 LB 实例组创建一个新的放置组规则，要求组内的负载均衡虚拟机优先位于集群中的不同主机（尽量保证 LB 虚拟机分散放置）；  
* 自动调度一次，将虚拟机调度至不同物理主机。

### **编辑**

两个 LBI 组成 LBG 后，如其中一个 LBI 编辑了如下内容，需提示用户编辑另一 LBI 中的配置信息

* 配置类型（只支持提升）  
* 是否支持 VPC 网络（仅当 LBG 中两个 LBI 都支持 VPC 网络时，才可以在关联网络时选择 VPC 网络）

### **健康状态**

除创建时填写的内容外，对 LBG 中的每个 LBI，需单独展示如下信息：

* LBI 健康状态：分别展示两个 LBI 的健康状态，正常/异常。  
* LB VM 健康状态：正常/异常。 🟡 2025-06-26 增加异常提示  
* 活动状态：只对 LBG 中当前实际运行虚拟服务的 LBI，展示为「活动」状态  
  * 正常情况下，主 LBI 为「活动」状态；  
  * 如系统检测到主 LBI 异常，自动切换为备用 LBI 提供服务，此时备用 LBI 为「活动」状态；  
  * 故障恢复后，如主 LBI 的健康状态恢复为「健康」，可自动切换回活动状态，或由用户手动执行切换。  
* 界面提示  
  * 如备用 LBI 为LBI 为「活动」状态，且主 LBI 的健康状态为「健康」，则提示用户主 LBI 已恢复健康，可将其切换回活动状态。  
  * 如主 LBI 为活动状态，且健康状态为「健康」，备 LBI 为「异常」，则提示用户备用 LBI 为异常状态，请尽快恢复。  
  * 如存在任一 LB VM 异常，提示：%主/备% LB 实例中，存在 %N% 个 LB 虚拟机故障，请尽快修复。🟡 2025-06-26 增加异常提示

### **故障切换**

#### 手动切换

* 支持将当前处于非活动状态，且健康状态为“健康”的 LBI 切换为活动状态；  
* 切换前，如活动 LBI 中存在任一虚拟服务，切换时需提示，切换过程中将有短暂的网络中断，建议在业务空闲时操作。

手动切换时，需用户二次确认，知晓切换过程中会有短暂的网络中断，并确认执行切换。

#### 自动切换

* 检测到当前的活动 LBI 的健康状态为「异常」后  
  1. 如另一 LBI 为「健康」状态，则自动将此 LBI 上的全部虚拟服务及 vip 切换至非活动 LBI；  
  2. 如另一 LBI 为「异常」状态，则不触发故障切换，给出报警信息。  
* 如当前备用 LBI 为「活动」状态，检测到主 LBI 恢复健康后，不会自动切回。（切换机制见[基础设定](#基础设定)）。

### **删除**

* 前提条件：已删除 LBG 中的所有虚拟服务。  
* 删除操作：  
  * 不允许单独删除 LBG 中的一个 LBI；  
  * 删除 LBG 时，仅解除主备 LBI 的关联关系，不会同时删除主、备 LBI，两实例仍可以作为独立 LB 实例继续使用。

# 服务器池组

## **基础设定**

1. 支持将 2 个已创建的服务器池关联为主备关系，组成服务器池组（Server Pool Group）。  
2. 此版本只支持主备模式，即仅当主服务器池中的所有后端服务器均不健康时，才切换至使用备服务器池。暂不支持主备服务器池层面同时工作，提供负载均衡。

## **故障切换**

🟡 2025-02-18 调整切换方式，[Slack 讨论](https://smartx1.slack.com/archives/C045HGJ7YNA/p1739867143804079?thread_ts=1739866976.793079&cid=C045HGJ7YNA)

对于已经关联到 vs 中的服务器池组，支持在 VS 中根据自动检查结果自动进行故障切换，或用户手动切换 VS 当前使用的活动服务器池。此处故障切换均为在 VS 中的操作，如多个 VS 使用相同的服务器池组，每个 VS 中的活动池可能不同。

### **故障场景**

1. 已配置主备 LBI，且发生集群整体故障时，切换方式及顺序如下：  
   1. 发现主 LBI 故障后，会自动将所有 vs 迁移至备 LBI；  
   2. 如 vs 配置了 Pool Group，健康检查发现主服务器池故障后，也将自动切换至备服务器池；  
   3. 用户修复主集群的异常，恢复 LBI 及 Pool 中成员的运行状态；  
   4. 过程中对 Pool Group 中的所有成员持续进行健康检查，主服务器池恢复后，不自动切换回主服务器池，原因如下：  
      * 可能由于修复期间的网络重启，会短时间恢复健康后再次故障，可能频繁出发故障切换；  
      * 与 LBI 的切换方式保持一致，目前 LBI 不会自动回切，如先回切服务器池，可能导致短时间内的流量绕行。  
   5. 主集群整体恢复后，用户手动执行 LBI 的切回，切回 LBI 后，如 vs 配置了 Pool Group，也尝试恢复使用主服务器池，原因如下：  
      * 大部分场景下，主服务器池和主 LBI 在同一物理集群中，同时恢复可避免流量绕行；  
      * 如不自动切回主服务器池，依靠用户手动切回，在 vs 数量较多的情况下，需要逐个操作，较为复杂。  
      * 说明：此时仅为尝试恢复，如果主服务器池健康状态仍为不健康，会继续使用备服务器池。因此即使主 LBI 和主服务器池不在同一集群，也不会造成异常。  
2. 未配置主备 LBI，或 LBI 无异常，仅服务器池自身异常  
   1. 健康检查发现主服务器池故障后，自动切换至备服务器池；  
   2. 用户修复主 Pool 的成员；  
   3. 过程中对 Pool Group 中的所有成员持续进行健康检查，主服务器池恢复后，不自动切换回主服务器池，原因同上；  
   4. 用户通过健康检查观察到主 Pool 中的成员状态恢复后，手动切回主服务器池。此时仅为单个服务器池的切换，操作简单，用户手动执行可避免发生频繁切换。

### **自动/手动切换方式**

基于上述场景，对服务器池的切换方式总结如下：

* 自动切换：仅在如下三种情况下，触发自动切换  
  * 主服务器池故障：根据健康检查结果，当主服务器池中所有后端服务器（包括备用成员）均为「不健康」状态时，自动切换至备服务器池。  
  * 备服务器池故障：**如果当前正在使用备服务器池**，检查到备服务器池中所有后端服务器（包括备用成员）均为「不健康」状态，且主服务器池中任一成员为「健康」时，自动切换至主服务器池。  
  * 跟随 LBI 切换，满足如下条件时（多条之间为“且”的关系），将使用主/备服务器池，切换 LBI 时提示用户：执行切换后，对于配置了服务器池组的虚拟服务，也将尝试恢复为使用主/备服务器池。  
    * 用户手动触发 LBI 切换；  
    * ~~将主 LBI 切换为活动实例；~~  
    * vs 已配置了 Pool Group；  
    * 目标 Pool 中至少存在一个成员（包括备用成员）状态为「健康」。

| 说明：如当前活动状态的服务器池未关联健康检查监视器，则始终将所有成员视为可用状态，不触发自动切换。🟡 2025-02-25 调整健康检查为可选项 |

# 虚拟服务

## **基础设定**

此需求中，虚拟服务本身的设定并无改变，但需要配合 LBG 和服务器池组的调整进行如下适配：

* 主备 LBI  
  * 每个虚拟服务需要属于一个 LBI 或 LBG。  
  * 虚拟服务属于 LBG 时，与客户端通信的网络，和与服务器池通信的网络，均需要选择 LBG 中，主备 LBI 关联的一对 VLAN 或 VPC 网络。  
* 主备服务器池：在虚拟服务中，可以关联一个服务器池或一个服务器池组。  
* 服务器池主备和 LBI 主备机制之间没有依赖关系，即无论虚拟服务属于 LBI 还是 LBG，都可以选择关联独立的服务器池或服务器池组。

  * 如与服务器通信的网络为 VLAN 网络，要求此网络（或主备 LBI 中的一对 VLAN 网络）均与服务器池/服务器池组中的所有成员（包括备用成员）三层连通。  
  * 如与服务器通信的网络为 VPC 网络，要求此 VPC 子网（或主备 LBI 中的一对 VPC 子网）均与服务器池/服务器池组中的所有成员（包括备用成员）处于同一个 VPC 内，可以位于相同子网或不同子网。

# 相关概念

* LB 实例（Load Balancing Instance，LBI）：表示一组（2-4 个）用于部署 LB 服务的虚拟机。

   对应实现文档中的 LB Group、LB 集群。

  说明：Everoute 2.0.0 版本中，每个 Everoute 服务中只支持部署一组此类虚拟机，因此没有专门进行命名，均以「负载均衡虚拟机」代称。从 Everoute 3.4.0 版本开始，支持部署多组此类虚拟机，也支持在两组虚拟机之间建立主备关系，因此为方便理解、避免歧义，需要一个统一的概念名词。

*  LB 实例组（Load Balancing Instance Group，LBG）：两个 LB 实例建立主备关系后，组成一个 LB 实例组，用于提供实例的高可用。

  对应实现文档中的 AE Group。

# 背景

目前每个 Everoute 服务中，支持部署一组（2-4 个） LB 虚拟机，以提供负载均衡服务。后续会通过 LB 支持 VPC 内的负载均衡，并扩展 Everoute 整体的高可用方案，同时也将在企业云中引入 LB 能力，支持多租户下的负载均衡。因此一组 LB 虚拟机从隔离性和流量负载上都难以继续满足要求。

在此背景下，需要增加支持多组负载均衡虚拟机，并可以通过两组虚拟机组成主备，增强 LB 的高可用能力。

此文档针对高可用场景，重点分析各种可能的情况和预期行为。

# 高可用级别

根据不同的主备切换粒度，存在如下三类高可用级别。

说明：此处仅针对 LB 实例、虚拟服务、服务器池的主备切换进行讨论，暂不考虑其所属的 SMTX 集群位置。

**讨论结论：**通过 LB 实例主备 \+ 服务器池主备已经可以满足大部分场景。为降低用户配置的复杂度，**在 ER 3.4 版本中，暂不支持虚拟服务级别高可用**，即不支持 LBG 为“主-主”模式，仅支持主备模式，LBI 故障时对所有虚拟服务统一迁移至备用 LBI。

## **服务器池级别**

主备服务器池的功能，不依赖 LBI 级别的主备，仅根据服务器池本身的健康状态进行判断和切换，因此无论 LBI 级别是否配置了主备，都支持使用。

### **配置 LBI（无高可用）**

主备两个服务器池组成一个服务器池组（Pool Group），虚拟服务可以关联到某个 Pool Group。未发生故障时，使用主服务器池（Pool1）提供服务。当检测到 Pool1 中的所有成员均不健康时，切换为使用 Pool2。

| 正常 | RS 故障 |
| :---: | :---: |
| ![][image13] | ![][image14] |

###  **配置 LBI Group**

主备服务器池结合主备 LBI 使用时，每个 LBI Group 下可以关联多个虚拟服务（此处仅以关联一个虚拟服务为例）。

* 服务器池故障  
  1. 主 LBI 中的 LB 虚拟机正常，但检测到 Pool1 中所有 RS 的状态异常。  
  2. 切换至使用 Pool 2 中的服务器成员。  
  3. 由于 LBI 正常，继续使用主 LBI 中的 LIP 与 Pool2 通信。此时如果主 LBI 和 Pool2 不在同一 SMTX 集群（或不在同一数据中心）内，可能发生流量绕行。

| 正常 | RS 故障 |
| :---: | :---: |
| ![][image15] | ![][image16] |

* LBI 故障  
  1. 控制面检查到主 LBI 中的所有 LB 虚拟机均故障。  
  2. 触发「LBI Group 级别」故障切换，将主 LBI 中的所有虚拟服务统一切换至备 LBI。  
  3. 此时使用备 LBI 的 LIP 与 Pool 通信并进行健康检查。  
  4. 如 Pool 1 无异常，则不触发切换，此时如果备 LBI 和 Pool1 不在同一 SMTX 集群（或不在同一数据中心）内，可能发生流量绕行。

| 正常 | RS 故障 |
| :---: | :---: |
| ![][image15] | ![][image17] |

* LBI 故障 \+ 服务器池故障  
  1. 控制面检查到主 LBI 中的所有 LB 虚拟机均故障。  
  2. 触发「LB 实例级别」故障切换，将主 LBI 中的所有虚拟服务统一切换至备 LBI。  
  3. 此时使用备 LBI 的 LIP 与 Pool 通信并进行健康检查。  
  4. 检测到 Pool1 中所有 RS 的状态异常。  
  5. 切换至使用 Pool 2 中的服务器成员。

# 场景说明 {#场景说明}

高可用相关的场景分析中，均不考虑管控面故障，假设集群均由同一个 CloudTower 管理，以及同一个 Everoute 服务管理。管控面高可用单独通过 CloudTower HA 及 Everoute Controller 的跨集群部署方案解决。

## **SMTX 集群内高可用**

* 部署方式：  
  * 主备 LBI 均部署在同一个 SMTX 集群中，分布在不同节点，**提供节点级别的高可用**。

    由于一个 LBI 中最多包含 4 个 LB 虚拟机，每个虚拟服务可由其中 2 个虚拟机提供主备；因此一个 LBI 组内最多包含 8 个 LB 虚拟机，每个虚拟服务可由其中 4 个虚拟机提供主备。（示意图中以每个 LBI 包含 2 个 LB 虚拟机的情况为例）

  * 后端服务器（RS）均位于此集群内。  
* 非故障场景下：由 LBI \- A 中的两个 LB 虚拟机提供服务，所有虚拟服务都运行在这两个虚拟机中，LBI-B 仅为备用，不实际进行流量转发。  
* LBI 虚拟机故障  
  * 如 LBI-A 中的一个虚拟机故障，其中的虚拟服务将转为由另一个 LB 虚拟机提供服务。  
  * 如 LBI-A 中的两个虚拟机均故障，将转为通过 LBI-B 中的虚拟机提供服务。  
* 物理机故障  
  * 物理机上的 LBI 虚拟机仍按照上述方式处理  
  * 主机上运行的后端服务器将通过集群 HA 在其他节点重建  
    * 发生故障后，LB 健康检查发现此主机上的 RS 异常，不再将新的流量转发到此 RS 中  
    * 如 RS 所在虚拟机已启用 HA，将自动在集群中的其他节点重建，重建后后端服务器的 IP 地址不变  
    * 重建后且业务拉起后，LB 健康检查将检测到服务器恢复，继续将新的请求转发至此服务器即可。

说明：此场景下，VPC 网络和 VLAN 网络的高可用机制相同，VPC 场景下保证所有 RS 在同一个 VPC 内即可。

| ----- | ----- |
| ![][image19] | ![][image20] |

## **跨 SMTX 集群高可用**

* 两个位于不同站点的 SMTX 集群，互为主备。  
* 主备 LBI 分别部署在一个 SMTX 集群中，**提供集群级别的高可用**。

后端服务器（RS）存在如下三种部署方式。

### **VLAN 网络 \+ 业务多实例**

* 部署方式  
  * 多个后端服务器（RS）分别部署在两集群。  
  * 两集群中的 RS 配置为两个 RS Pool，网关可以不同。  
  * LBI 通过 VLAN 网络与 RS 通信，要求主备 LBI 均与两个 Pool 中的 RS 均三层互通。  
* 非故障场景下：  
  * 仅由主集群中的 LBI 提供服务，VIP 和 LIP 均在 LBI-A 关联的 VLAN 网络中。  
  * 两服务器池中的 RS 均正常运行，但仅由主集群中的 RS 进行流量负载分担（需在主备 Pool 的 RS 之间进行数据同步）。  
  * 通过 LBI-A 的 LIP 对两个服务器池（PoolA、PoolB）中的所有 RS 进行健康检查，预期主备 Pool 中的 RS 均健康。  
* 主集群整体故障：即，LBI-A 中的所有 LB 虚拟机均故障；且 Pool-A 中的所有 RS 均故障。  
  1. Everoute 控制面发现 LBI-A 故障。  
  2. 触发 LBI 故障切换，将其中的所有虚拟服务统一切换至备 LBI-B，所有 VIP 均漂移到 LBI-B 关联的 VLAN 网络中。  
  3. 通过 LBI-B 的 LIP 与两个 Pool 中的 RS 通信并进行健康检查。  
  4. 发现 PoolA 中所有的 RS 均不健康。  
  5. 触发 Pool 的切换，使用 PoolB 中的 RS 进行流量负载分担。

| 非故障场景 | 集群故障场景 |
| ----- | ----- |
| ![][image21] | ![][image22] |

### **VLAN 网络 \+ 复制计划**

* 部署方式  
  * 所有后端服务器均部署在主集群。  
  * 借助「备份与容灾」产品配置复制计划，备集群中存在副本虚拟机。  
  * 将主集群中的 RS 配置为 PoolA，备集群中的副本虚拟机位置为 PoolB，网关可以不同，共同组成服务器池组。  
  * LBI 通过 VLAN 网络与 RS 通信，要求主备 LBI 均与两个 Pool 均三层互通。  
* 非故障场景下：  
  * 仅由主集群中的 LBI 提供服务，VIP 和 LIP 均在 LBI-A 关联的 VLAN 网络中。  
  * 通过 LBI-A 的 LIP 对两个服务器池（PoolA、PoolB）中的所有 RS 进行健康检查，预期 PoolA 中的 RS 均健康，PoolB 中的 RS 由于对应的是副本虚拟机，此时未关机状态，健康检查结果为“不健康”（此场景中不健康为预期现象，无需用户关注）。  
  * 由主集群中的 RS 进行流量负载分担，通过复制计划定期进行数据同步。  
* 主集群整体故障：即，LBI-A 中的所有 LB 虚拟机均故障；且 Pool-A 中的所有 RS 均故障。  
  1. Everoute 控制面发现 LBI-A 故障。  
  2. 触发 LBI 故障切换，将所有虚拟服务统一切换至备 LBI-B，所有 VIP 均漂移到 LBI-B 关联的 VLAN 网络中。  
  3. 由于 RS 配置了复制计划，此时用户将触发故障转移，拉起备集群中的副本虚拟机。  
  4. 通过 LBI-B 的 LIP 与两个 Pool 中的 RS 通信并进行健康检查  
     1. 发现 PoolA 中的 RS 均不健康。  
     2. 由于故障转移需要一定的拉起时间，初期可能 PoolB 中的 RS 也均不健康。  
     3. 副本虚拟机成功拉起后，LBI-B 可以检测到 PoolB 中的 RS 为健康状态。  
  * 触发 Pool 的切换，使用 PoolB 中的 RS 进行流量负载分担。

| 非故障场景 | 集群故障场景 |
| ----- | ----- |
| ![][image23] | ![][image24] |

### **VPC 网络 \+ 复制计划**

* 部署方式  
  * 主备集群之间通过 VPC 将业务网络拉通。  
  * 所有后端服务器均部署在主集群。  
  * 借助「备份与容灾」产品配置复制计划，备集群中存在副本虚拟机，副本虚拟机与原虚拟机在同一个 VPC 子网内，且 IP 地址相同。  
  * 将 RS 配置为 Pool（无需配置 Pool Group）。  
  * LBI 通过 VPC 网络与 RS 通信。  
* 非故障场景下：  
  * 仅由主集群中的 LBI 提供服务。  
  * 通过 LBI-A 的 LIP 对服务器池中的所有 RS 进行健康检查，预期 RS 均健康。备集群中的 RS 由于与原虚拟机 IP 地址相同，无需在 Pool 中单独配置。  
  * 由主集群中的 RS 进行流量负载分担，通过复制计划定期进行数据同步。  
* 主集群整体故障：即，LBI-A 中的所有 LB 虚拟机均故障；且 Pool 中的所有 RS 均故障。  
  1. Everoute 控制面发现 LBI-A 故障。  
  2. 触发 LBI 故障切换，将所有虚拟服务统一切换至备 LBI-B，所有 VIP 均漂移到 LBI-B 关联的网络中。  
  3. 由于 RS 配置了复制计划，此时用户将触发故障转移，拉起备集群中的副本虚拟机。  
  4. 通过 LBI-B 的 LIP 与 Pool 中的 RS IP 地址通信并进行健康检查  
     1. 由于故障转移需要一定的拉起时间，初期原虚拟机已故障，副本虚拟机拉起之间，可能 Pool 中的 RS 均不健康。  
     2. 副本虚拟机成功拉起后，LBI-B 可以检测到 Pool 中的 RS 为健康状态，可以正常进行流量转发。

| 非故障场景 | 集群故障场景 |
| ----- | ----- |
| ![][image25] | ![][image26] |

## **双活 SMTX 集群高可用**

* 部署方式  
  * SMTX 双活集群的两个可用域分别位于不同站点（数据中心）。  
  * 主备 LBI 分别在优先可用域和次级可用域中，**提供可用域级别的高可用**。  
  * 后端服务器（RS）均部署在双活集群中，非故障场景下均运行在有限可用域。  
    1. 如两可用域之间业务网络二层连通，可直接通过 VLAN 网络通信，配置一个 Pool，与「SMTX 集群内高可用」场景类似。  
    2. 如两可用域之间业务网络三层连通，可通过 VPC 将网络拉通，配置一个 Pool，与「跨 SMTX 集群高可用」中的「VPC 网络」场景类似。  
* 非故障场景下：  
  * 仅由优先可用域中的 LBI 提供服务。  
  * 所有 RS 均运行在优先可用域中，进行流量负载分担，数据通过存储网络在两可用域之间实时同步。  
  * 通过 LBI-A 的 LIP 对服务器池中的所有 RS 进行健康检查，预期 RS 均健康。  
* 主可用域整体故障：即，LBI-A 中的所有 LB 虚拟机均故障；且 Pool 中的所有 RS 均故障。  
  1. Everoute 控制面发现 LBI-A 故障。  
  2. 触发 LBI 故障切换，将所有虚拟服务统一切换至备 LBI-B，所有 VIP 均漂移到 LBI-B 关联的网络中。（需为所有 LB 虚拟机关闭“虚拟机 HA”，保证在可用域故障时 LB 虚拟机不在次级可用域中重建。）  
  3. 通过 LBI-B 的 LIP 与 Pool 中的 RS IP 地址通信并进行健康检查  
     1. 由于 HA 需要一定的拉起时间，初期虚拟机已故障，HA 重建完成之前，可能 Pool 中的 RS 均不健康。  
     2. RS 在次级可用域重建完成后，LBI-B 可以检测到 Pool 中的 RS 为健康状态，可以正常进行流量转发。
