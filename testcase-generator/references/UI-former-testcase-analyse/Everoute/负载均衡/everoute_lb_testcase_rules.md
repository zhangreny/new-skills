# Everoute 负载均衡 UI Testcase Rules

## 必须遵守

- 顶层优先按对象组织，例如 LB / LBI / LBG / 虚拟服务 / 服务器池 / 服务器池组 / 事件审计与任务 / 权限管理 / 异常限制。
- 第二层优先按页面或动作组织，例如部署 Everoute 服务、启用 LB、创建、编辑、详情、关联网络、解除关联、删除。
- 对象层次深的场景必须继续拆到“子模块 / 通信方向 / 网络类型 / 字段组 / 配置 IP / 状态”后再落 `[C]`。
- 只要版本、VPC 开关状态、网络模式、通信方向、主备一致性、池/池组类型不同导致展示或校验不同，就必须显式拆层。
- LB 相关需求通常有强矩阵展开，不允许把“对象 × 动作 × 版本/模式 × 结果”过早压缩成单条 case。
- 事件审计、权限管理、异常限制属于 LB 固定块，默认必须纳入。

## 常见分层模式

- 对象 -> 页面/动作 -> 子模块 -> 状态/条件 -> `[C]`
- 对象 -> 网络设置/关联网络 -> 通信方向 -> 字段组 -> 校验或结果 -> `[C]`
- 对象 -> 创建/编辑 -> 开关或模式 -> 网络类型/字段 -> `[C]`
- 对象 -> 详情 -> 卡片或 tab -> 展示项 -> `[C]`
- 对象 -> 事件审计与任务 / 权限管理 / 异常限制 -> 动作或限制类型 -> `[C]`

## 展开规则

- 高优先级展开轴：对象类型、动作类型、版本、VPC 支持状态、网络类型、通信方向、主备关系、容量状态、结果。
- 只要是网络配置类场景，优先按“网络类型 -> 字段组 -> 校验/分配方式”展开。
- 只要是主备场景，主实例、备实例、主备一致、主备不一致必须显式区分。
- 只要是关联网络/网络组场景，与客户端通信、与服务端通信、客户端+服务端通信应优先保留为中间层。
- 只要存在自动分配/手动分配/部分自动部分手动等模式差异，应在字段或配置组下继续展开。
- 许可类 LB 需求，优先使用“场景 / 关联功能 / 许可子类 / 环境对象 / 容量状态 / 结果”的组合模式。

## 禁止项

- 禁止把 LBI、LBG、虚拟服务、服务器池等不同对象混在同一动作分组下。
- 禁止把开关显隐、字段显隐、字段校验、提交结果压成一条 case。
- 禁止把主备关系、通信方向、网络类型等关键轴折叠成“配置正确”。
- 禁止遗漏权限管理、事件审计与任务、异常限制。
- 禁止在长层级对象里为了节省篇幅而突然减少中间分组深度。
