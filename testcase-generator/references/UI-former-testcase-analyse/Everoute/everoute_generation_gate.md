# Everoute Testcase Generation Gate

## 规则加载

- 必须加载 `everoute_common_testcase_rules.md`。
- 必须按主需求所属产品加载一个主产品规则文件。
- 如果当前需求同时包含其他产品的独立 UI 变更，可额外加载对应产品规则作为补充规则。
- 主产品规则决定顶层分组方式，补充产品规则只影响相关子树。

## 规则执行回合

在 Step6、Step7、Step8 中都必须显式完成一次“规则执行回合”，至少记录以下内容：

- 已加载的规则来源
- 当前采用的分层模板
- 当前采用的主展开轴
- 检查过的禁止项
- 发现的违规项与修正结果

## Step6 门禁

- 只允许输出可复用分组维度，不允许输出 testcase 级结论。
- 必须控制在 6-10 个核心维度内，除非材料本身明显跨多个产品或多个大型对象。
- 如果维度值已经带有完整动作、前提和结果，则说明下沉过度，必须回退。
- 如果维度只剩“正常 / 异常 / 边界”这类抽象词，则说明展开不足，必须补充。

## Step7 门禁

- Step6 中保留的每个核心维度，要么在 Step7 组合展开中出现，要么显式说明为何不展开。
- Step7 只允许输出组合，不允许出现 `[C]`。
- 不允许把不同版本、不同状态、不同权限、不同结果的组合提前合并。
- 如果一个叶子节点同时表达两个以上独立结果，应继续拆开。

## Step8 门禁

- Step8 的 testcase 必须承接 Step7 的组合，不允许跳过组合阶段重新自由生成。
- `[C]` 必须都落在最细层。
- 不允许把“新集群成功 / 旧集群报错”“有权限可编辑 / 无权限只读”压成一条 case。
- 主产品固定块不得遗漏：
  - 运维：版本差异 / 事件审计与任务 / 权限管理 / 异常操作限制
  - VPC：列表 / 创建 / 编辑 / 删除 / 详情 / 事件审计 / 权限控制
  - LB：对象层次 / 网络配置层次 / 事件审计与任务 / 权限管理 / 异常限制
  - DFW：若本次涉及，则必须保留 DFW 专属文案、关联或规则分组

## 缩水检查

- 是否出现顶层分组稳定，但中层变量轴明显丢失
- 是否后半段来源变粗
- 是否后半段层级深度明显降低
- 是否出现大场景下直接挂“成功 / 失败”这种未展开节点
- 是否遗漏版本差异、权限、异常限制、文案变化、事件审计等高优先级固定块
