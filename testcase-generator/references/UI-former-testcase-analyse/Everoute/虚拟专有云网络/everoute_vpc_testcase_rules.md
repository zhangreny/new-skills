# Everoute VPC UI Testcase Rules

## 必须遵守

- Step6 先读取 `everoute_vpc_product_knowledge.md`，用其识别 VPC 对象树、tab/section 结构、共享/非共享、主备、多 CIDR 和引用限制，再决定产品资料与历史样本。
- 顶层优先按“列表 / 创建 / 编辑 / 删除 / 详情 / 事件审计 / 其他相关改动 / 权限控制”组织。
- VPC 产品中，列表展示、详情页 tab、表单字段、路由或关联对象联动，必须保持独立分组，不允许混写。
- 只要详情页存在 tab 或附属对象（如路由表、对等连接、子网、TEP IP 等），必须先拆 tab/对象层，再落 case。
- 只要字段显隐、可编辑性、过滤范围、默认值、错误提示不同，就必须拆成字段级分组。
- 权限控制、事件审计、删除限制、关联限制属于 VPC 高频固定块，默认必须纳入。
- 若运维主线需求中出现 `GatewayVM`、`VPC 集群关联`、`TEP IP 池` 这类 subsection，仍按 VPC 对象展开，不可并入泛化的“系统服务网络配置”。

## 常见分层模式

- 业务对象 -> 列表 / 创建 / 编辑 / 删除 / 详情 -> 字段或校验点 -> `[C]`
- 详情 -> tab -> 展示项或关联对象 -> `[C]`
- 业务对象 -> 事件审计 -> 动作类型 -> `[C]`
- 业务对象 -> 权限控制 -> 可编辑 / 只读 -> `[C]`
- 业务对象 -> 其他相关改动或限制 -> 关联对象/引用对象 -> `[C]`

## 系统服务网络编辑补充

- `VPC 集群关联 -> TEP IP` 场景必须保留：主机名称升序展示、头部批量填充、留空自动分配、自动分配顺序与展示顺序一致。
- `VPC 集群关联` 的二次确认必须继续拆出顶部提示框、`VDS` 分区、`TEP IP` 分区和自动分配场景，不能只写“确认弹窗正确”。
- `TEP IP 池` 切换必须保留：变更后清空已有 `TEP IP`、placeholder 为自动分配、自动创建网络选项在编辑态不可选。
- `TEP IP 池` 编辑必须保留：子网掩码可编辑、子网掩码不可为空、网关与地址池同网段校验，以及“编辑规则与创建一致”的继承关系。
- `GatewayVM` 网络编辑必须把 `管理网络` 与 `TEP 网络` 分开，并继续保留批量编辑子网掩码/网关、二次确认按修改项分区显示、未修改的 VM/字段不出现在确认内容中。

## 展开规则

- 高优先级展开轴：页面动作、字段显隐、过滤逻辑、详情 tab、关联对象状态、权限、删除限制。
- 长名称展示、超长文案、链接跳转、详情页复用展示，通常应放在页面或 tab 分组下统一展开。
- 只要字段的前置选择会影响后续字段可选范围或禁用态，就必须把这种前后依赖关系显式展开。
- 对等连接、多网关、路由表、TEP IP 等带“关联对象”的场景，优先按对象关系展开，再看结果。
- 对文案调整类需求，必须单独保留“文案变化”结果轴，不要并入普通成功/失败场景。
- 涉及 `关联集群 / TEP IP 池 / GatewayVM` 的系统服务网络编辑时，必须显式保留 `VDS + TEP IP 池 + 每主机/每 VM IP + 主机名称排序 + 自动分配方式`，不能省略为抽象“网络编辑成功”。
- Step7/8 必须把历史样本中的“多个外部子网组”“多 CIDR”“反复切换活动网关”视为继续拆开的线索，而不是直接继承为单条 `[C]`。

## 测试资源类型

### VPC 对象资源

- VPC
- 关联集群
- 子网
- 边缘网关 / 边缘网关组
- 外部子网 / 外部子网组
- 路由表
- 安全服务
- 浮动 IP
- TEP IP 池
- TEP IP
- VDS
- 对等连接

### VPC 系统服务网络资源

- 管理网络 / TEP 网络
- 每主机 TEP IP / 每 VM TEP IP
- 主机名称排序
- 自动分配 / 手动填写
- 变更后清空 / placeholder 提示
- 新增节点 / 既有节点

### VPC 许可资源

- 许可对象：关联集群
- 许可方式：按关联集群的物理 CPU 插槽
- 额度状态：充足 / 刚好 / 不足 / 已满
- 结果资源：关联成功 / 关联失败 / 编辑受限 / 文案变化

### VPC 引用与限制资源

- 资源属于当前 VPC / 不属于当前 VPC
- 资源已被引用 / 未被引用
- 可删除 / 不可删除
- 过滤范围
- 关联限制

### 权限与审计资源

- 权限角色
- 审计动作
- 删除限制

## 资源展开提示

- 若当前需求涉及 VPC 许可或关联集群，必须保留 `许可对象 + 许可方式 + 关联集群状态 + 结果`。
- 只要当前场景涉及资源选择、过滤范围、引用关系或删除限制，就必须把这些资源关系显式写出来。
- 在 LB/VPC 联动任务中，VPC 资源主要作为“补充资源层”作用于相关子树，不得反客为主。
- 若当前需求涉及 `GatewayVM / VPC 集群关联 / TEP IP 池`，必须把二次确认分区、自动分配顺序、切换池清空和创建规则继承一起前置到蓝图中，而不是等到 Step9 才补。

## 禁止项

- 禁止把列表页、详情页、弹窗页的检查点混写在一个分组里。
- 禁止把“字段展示正确 / 字段可编辑 / 字段过滤正确 / 字段报错正确”压成一条 case。
- 禁止遗漏事件审计、权限控制、删除限制、关联限制这些 VPC 典型固定块。
- 禁止在有 tab、stepper 或关联对象时跳过中间层直接落 `[C]`。
- 禁止把具体路由、对等连接、TEP IP 的关联前提直接压缩成抽象“配置正确”。
- 禁止把共享/非共享、主备、多 CIDR、过滤范围和结果归属混成一个泛化的“网络配置正常”。
- 禁止把 `主机名称顺序 + 批量填充 + 自动分配 + 切换池清空` 合并成抽象“TEP IP 配置正确”。
- 禁止把 `管理网络修改 + TEP 网络修改 + 二次确认内容` 压成一条 case。
