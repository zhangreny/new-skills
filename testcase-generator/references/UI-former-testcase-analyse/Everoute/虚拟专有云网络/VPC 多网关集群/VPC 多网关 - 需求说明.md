# 概述

## **相关概念**

* **边缘网关**  
  * VPC 服务的边缘网关虚拟机，它具有到 VPC 的下行链路连接，以及到物理网络的外部连接，作用是实现 VPC 与物理网络的三层互通。如果 VPC 需要与数据中心、公网等外部网络进行通信，则必须先在服务中创建边缘网关。  
  * Everoute VPC 中，每个边缘网关由 2 个边缘网关虚拟机组成。  
* **网关服务**：为 VPC 子网实际提供流量转发的 NAT 网关、路由网关、L2 网关，广义范围内也可以包含浮动 IP，也称为“网关实例”。

## **背景**

目前支持为 VPC 部署边缘网关，提供 VPC 内与外部链接的网关服务。但目前仅支持为每个 VPC 服务部署一个边缘网关，由 2 个边缘网关虚拟机组成，虚拟机之间通过心跳机制实现高可用的主备容灾。

VPC 的一个重要使用场景是为业务网络提供高可用，用户通过多个物理环境部署可用域、集群、数据中心级别的业务容灾及高可用方案时，可以通过 VPC 打通多个物理环境之间的网络。但目前仅支持 1 个边缘网关，且提供服务的 2 个边缘网关虚拟机需要部署在同一个集群中，如果发生集群或数据中心级别的故障，需要先手动恢复/拉起边缘网关，才能继续提供服务，整体恢复时间较长，难以满足用户需求。

因此需支持多网关的部署模式，在一个集群或多个集群中部署多对边缘网关虚拟机。

## **目标**

* 负载分担：部署多个边缘网关，为同一 VPC 或不同 VPC 提供网关服务，分担与外部通信的流量负载。  
* 网络高可用：提供节点、可用域（双活场景下）、集群、数据中心层级的网络高可用及容灾能力。故障场景下，虚拟机通过高可用、备份或复制机制被重新拉起后，可以在尽量不变更网络配置的情况下尽快恢复连接，缩短 RTO 时间。

## **场景分析**

### **流量负载分担**

可在集群内或跨集群部署多个边缘网关，分别为不同的 VPC、VPC 子网或网关服务提供流量转发能力。

可选择如下方式：

* 按 VPC 划分：如每个 VPC 由不同部门管理，希望隔离不同 VPC 的网关需求，可为每个 VPC 单独部署一个边缘网关，仅对此 VPC 提供服务，此 VPC 内所有网关服务均由这个边缘网关提供。

  ![][image1]

* 按 VCP 子网划分：如 VPC 内对外流量较多，可通过子网划分，即为每个子网单独部署一个边缘网关，仅对此子网提供服务。

  注意：跨集群场景下，如当前子网内大部分业务虚拟机均处于集群 1，也建议将其专用的边缘网关部署在集群 1，避免流量绕行。

  ![][image2]

* 按网关服务划分：可以选择为每类网关服务部署一个边缘网关，例如单独划分一个边缘网关，专门用于为所有 VPC 提供 LB 服务。

  ![][image3]

由于边缘网关与 VPC/ VPC 子网之间可通过网关实例和路由表进行灵活关联，因此支持多边缘网关后，可根据需求灵活配置不同网关的负载情况，不仅限于上述场景。(需要明确边缘网关的数量上限，总数、每 VPC？每集群？每子网？)

### **增强集群内高可用** {#增强集群内高可用}

在 [VPC 网关高可用  - 场景分析](https://docs.google.com/document/d/12i8vwvH_dDWE4Mm7GF_gVo9mIIaRycJoOv96R2slw0w/edit#heading=h.wfhi0u4x2tzo) 中，对同集群、跨集群及双活集群在各类网络条件下的高可用方案进行了分析。

此场景对应的网络条件为：**集群内的管理网络、TEP 网络及外部网络均 L2 互通。**

目前每个边缘网关中仅包含 2 个边缘网关虚拟机，如网关虚拟机所在的 2 个物理节点同时发生故障，则数据面将无法使用。目前的方式下，需要借助虚拟机 HA 机制将网关虚拟机在其他节点拉起，此过程较长，可能导致长时间的数据面通信中断。

因此可在集群中创建 2 个边缘网关，对应 4 个边缘网关虚拟机，分别部署在集群的 4 个物理节点中，构成一组主备边缘网关，由于外部网络 L2 互通，因此两网关中可对应相同的外部子网网段。如主边缘网关的 2 个虚拟机同时故障，可以触发主备切换，通过备用边缘网关继续提供对外通信服务。

架构如下：

* 非故障期  
  * 通过 Edge1 为此 VPC 中的所有虚拟机提供网关服务；  
  * Edge2 作为此 VPC 的备用网关，此边缘网关及其外部子网均仅作为备份使用，非故障期不承载任何业务。  
* 故障期  
  * 管理面：CloudTower 通过虚拟机 HA 在集群其他节点拉起（后续结合 Tower HA 功能，可提供跨站点的管理面高可用）；  
  * 控制面：如 Everoute Controller 的故障节点未超过半数，控制面无需恢复；  
  * 业务虚拟机：通过虚拟机 HA 在集群其他节点拉起。  
  * 网络配置  
    1. VPC 内的 IP/子网均无需变化；  
    2. 数据面检测到 Edge1 故障后，自动切换为由 Edge2 提供网关服务；  
    3. 底层网络无需调整。

**配置建议：**同集群场景，或两集群/两可用域之间的外部网络 L2 互通，可以在主备网关配置完全一致的外部子网（例如图中两侧外部子网 CIDR 均为 192.168.1.0/24，为同一个虚拟机分配的浮动 IP 也均为 192.168.1.10），系统层面可保证备用网关及其外部子网在非故障期间不会用做其他用途，因此无需担心 IP 冲突。

* 优点是：  
  * 主备切换完成后对外提供服务的 DNAT 及 FIP 地址不变，客户端无需额外处理；  
  * 地址不变时后端可通过 IP 地址对已有连接进行 session 同步，保持提供服务的连续性。  
* 缺点是：对外部网络依赖较多，跨集群场景下，主备切换后可能仍需要网络管理员对硬件设备进行配置变更。

### **跨集群高可用** {#跨集群高可用}

此场景对应的网络条件为：**两集群/两可用域之间，管理网络及 TEP 网络可 L2/L3 互通，外部网络不通。**

可在两集群/两可用域中，分别创建一个边缘网关，对应不同的外部子网网段，并将两个边缘网关绑定为主备关系。

* 非故障期  
  * 通过 Edge1 为此 VPC 中的所有虚拟机提供网关服务；  
  * 业务虚拟机运行在集群 1，可以将部分业务运行在集群 2，但均通过 Edge1 提供出口网络，在集群 2 的业务将存在流量绕行；  
  * Edge2 作为此 VPC 的备用网关，此边缘网关及其外部子网均仅作为备份使用，非故障期不承载任何业务。（后续支持同宿同源、ECMP、动态路由等功能后，可以考虑备用网关在非故障期也可承载其他业务流量。）

  ![][image5]

* 故障期  
  * 管理面：需在集群 2 将 CloudTower 虚拟机拉起（后续结合 Tower HA 功能，可提供跨站点的管理面高可用）；  
  * 控制面：需在集群 2 将 Everoute Controller 拉起；（建议尽快讨论 Controller 的高可用机制改进，是否可参考 tower ha 的仲裁机制）  
  * 业务虚拟机  
    * 集群 2 的虚拟机继续提供服务；  
    * 集群 1 的虚拟机通过备份/复制恢复在集群 2 拉起，继续使用 VPC1 的子网。  
  * 网络配置  
    1. VPC 内的 IP/子网均无需变化；  
    2. 数据面检测到 Edge1 故障后，自动切换为由 Edge2 提供网关服务；  
    3. 在 Edge2 中通过 BGP 宣告新的路由规则，进行路由切换。  
    4. 底层网络无需调整。

![][image6]

* 故障恢复：通过如下两种方式之一将网关服务回切至主网关  
1. 系统检测到主网关恢复后，自动回切  
   1. 好处：如果非整个集群故障，仅 Edge 虚拟机故障，自动回迁可继续通过主网关提供服务。  
   2. 可能的问题是：如果主集群仅 Edge 虚拟机恢复， 业务虚拟机暂未回迁，自动回迁将导致一段时间的流量绕行，且如果集群处于不稳定状态（例如维护过程中），可能会多次触发迁移/回迁动作。  
2. （推荐）主集群所有服务恢复后，用户手动回切：对用户来讲可控性更高，但需要额外引入一次手动操作。

**配置建议：**跨集群场景下，如两集群/两可用域之间的外部网络位于不同的 L2 网络，需要配置不同的外部网段，以适应不同站点的不同外部网络配置。备用站点中可提前配置对应的外部子网网关，主备切换后无需对硬件设备进行配置变更。建议两侧使用对称的外部网段，例如图中两侧外部子网 CIDR 分别为 192.168.1.0/24 和 192.168.2.0/24，为虚拟机分配的浮动 IP 分别为 192.168.1.10 和 192.168.2.10。

但使用此方式，

* 主备切换后对外提供服务的 DNAT 及 FIP 地址将发生变化，客户端需访问新的地址。可在备用网关激活后，通过动态路由协议自动向外部网络宣告路由。  
* 外部子网对应不同的 CIDR，在 VPC 中为每个网关实例分配外部 IP 时，都需要对应分配主 IP 地址和备用 IP 地址。

### **双活集群高可用**

部署方式：在双活集群部署边缘网关时，需明确网关的可用域属性。

1. **主备网关** \- 位于**不同**可用域：  
   * 主备边缘网关分别部署在一个集群中的主可用域和次级可用域，提供可用域级别的高可用；  
   * 需通过必须放置组规则限制，网关虚拟机只允许位于对应可用域，当前可用域故障时，不在另一可用域触发 HA 重建，即始终保持主备边缘网关位于不同的可用域。  
2. **主备网关** \- 位于**同一**可用域：同[增强集群内高可用](#增强集群内高可用)场景，无需额外处理  
   * 主备边缘网关部署在一个可用域内的不同节点，提供节点级别的高可用；  
   * 需通过必须放置组规则限制，网关虚拟机只允许位于对应可用域，当前可用域故障时，不在另一可用域触发 HA 重建，即始终保持主备边缘网关位于同一可用域。  
3. **主备网关** \- 分别位于双活集群的一个可用域和另外一个独立集群：同[跨集群高可用](#跨集群高可用)场景，无需额外处理  
4. **独立网关** \- 两个网关虚拟机位于不同可用域：暂不支持  
   * 网关中的两个边缘网关虚拟机，分别位于优先可用域和次级可用域，提供可用域级别的高可用；  
   * 由于如下原因，暂不支持此方案：  
     * 网关服务使用的边缘网关虚拟机随机分配，不可避免会存在站点间的流量绕行；  
     * 对网络要求较高，只适用于可用域之间业务网络 L2 互通的场景；  
     * 每个可用域中只有一个边缘网关虚拟机，可靠性低于上述方案 1。  
5. **独立网关** \- 两个网关虚拟机位于同一可用域：3.0 版本中，已支持此方式  
   * 不指定的情况下，双活集群中的两个边缘网关虚拟机，默认均处于优先可用域。

下文仅基于上述部署方式 1 进行讨论：

* 外部子网：根据业务网络的连通情况决定配置方式  
  * 如可用域之间的外部网络 L2 连通，则主备网关上可配置相同的外部子网；  
  * 如可用域之间的外部网络 L3 连通，则主备网关上需在两个网段内配置对称的外部子网。  
* 故障恢复方式  
  * 业务虚拟机通过集群内的 HA 机制，自动在另一可用域的主机中拉起；  
  * 故障可用域内的边缘网关由于必须放置组限制，不会在另一可用域中拉起；  
  * 检测到可用域故障后，自动切换为，通过备用边缘网关提供服务。  
* 故障判定条件

  目前双活集群采用两可用域 \+ 仲裁节点的架构，根据节点及网络的故障情况，采取不同的恢复措施（具体参考[双活集群的故障处理](http://docs.fev.smartx.com/smtxos/6.0.0/best-practice-multiactive/best_practice_multiactive_77)）。

  以业务运行在优先可用域，且主边缘网关也位于优先可用域的场景为例，各类故障中，对主备网关的切换方式预期如下： 

| 故障场景 |  | 业务虚拟机 HA 重建方式 | 主备网关的切换预期 |
| :---: | :---- | ----- | ----- |
| 节点故障 | 优先用域的**部分**节点故障 | 在优先可用域的其他节点重建。 | 故障节点中的网关虚拟机，在优先可用域内其他节点通过 HA 重建； 主备网关不切换。 |
|  | 优先用域的**全部**节点故障 | 在次级可用域重建。 | 主网关中的网关虚拟机不触发 HA 重建； 流量切换至备用网关。 |
|  | 仲裁节点故障 | 不影响业务虚拟机。 | 不影响，不触发流量切换。 |
| 网络故障 | 优先可用域与仲裁节点间网络中断； 次级可用域与仲裁节点间网络连接正常 | 在次级可用域重建。 | 主网关中的网关虚拟机不触发 HA 重建； 流量切换至备用网关。 |
|  | 两可用域与仲裁节点间网络连接均中断。 | 不触发 HA 重建。 | 不触发流量切换。 |
|  | 两可用域之间的存储网络中断。 判定方式：ZooKeeper Leader 所在的可用域处于可用状态，另一可用域节点处于失连状态。 如果 ZooKeeper Leader 位于仲裁节点 C，则优先可用域 A 处于可用状态，次级可用域 B 处于失连状态。 | 可用状态的可用域中，虚拟机不触发 HA 重建。 失联状态的可用域中，虚拟机在另一可用域重建。 | 如主网关所在可用域为可用状态，不触发流量切换。 如主网关所在可用域为失联状态，流量切换至备用网关。  |

如暂不进行上述改造，在部署 VPC 网关时可暂不引入“可用域“的概念，只采用基础的方式，把双活集群作为一个集群考虑。

* 此时将存在上述 2、3、5 的部署方式；  
* 不特意指定可用域的情况下，网关虚拟机将始终处于优先可用域；  
* 优先可用域故障（或两可用域之间的存储网络故障/仲裁节点与优先可用域之间的网络故障）时，两个网关虚拟机将通过 HA 自动在次级可用域重建  
  * 如可用域之间的外部网络 L2 连通，即使用同一网段，网关重建后可继续提供服务。  
  * 如可用域之间的外部网络 L3 连通，即使用不同的业务网段，网关虚拟机触发 HA 重建后仍无法使用。

## **基础设定**

### **边缘网关**

* 边缘网关定义：每个边缘网关均由 2 个边缘网关虚拟机组成，此定义仍与 ER 3.0.0 中的设定保持一致。  
* 每个 Everoute 服务内，允许创建多个边缘网关（可 \> 2 个）  
  * 多个边缘网关之间：  
    * 可以相互独立，提供流量负载分担；  
    * 或，2 个边缘网关可以关联为主备关系，构成边缘网关组。  
  * 多个边缘网关可位于相同 SMTX 集群或不同集群，位于不同集群时，需管理网络和 TEP 网络连通。  
* 边缘网关组  
  * 可将 2 个边缘网关关联为主备关系。关联方式如下：  
    * ~~创建时新建 2 个边缘网关（共 2 \* 2 个边缘网关虚拟机），作为主备网关；~~  
    * 关联 2 个已有边缘网关，建立主备关系。  
  * 2 个边缘网关，可以位于两个 SMTX 集群，或同一集群中的不同物理节点。  
  * 可指定其中 1 个网关为主网关，一个为备用网关。  
* 故障切换  
  * 自动切换：检测到主网关故障时，自动将所有网关实例及其流量切换至其关联的备用网关。

| 故障恢复后不会自动切回，原因如下： 如果主集群中的边缘网关虚拟机已恢复， 但业务虚拟机暂未回迁，自动回迁将导致一段时间的流量绕行，且如果集群或边缘网关虚拟机处于不稳定状态（例如维护过程中），可能会反复触发主备切换动作。综合考虑，不支持自动切回，待主站点所有服务恢复后，用户手动回切，对用户来讲可控性更高。 |
| :---- |

  * 手动切换：支持手动对当前的所有网关实例及其流量切换至网关组中的另一边缘网关，使用场景如下  
    * 系统未自动检测到故障；  
    * 故障恢复后需切回。

### **外部子网**

* 每个外部子网都属于一个边缘网关；  
* 对于边缘网关组，可以对应一个或多个外部子网组  
  * 一个外部子网组中包含两个外部子网，互为主备；  
  * 从属关系为，主外部子网属于主边缘网关；备外部子网属于备边缘网关。  
* 外部子网组中的主 \- 备外部子网，可以共用相同的网络配置，或分别有不同的网络配置  
  * 共用网络配置  
    * 主 \- 备外部子网使用相同的 VLAN 网络；具有相同的 CIDR 块、网关信息；为各用途下分配的地址块均一致；  
    * 用户创建、编辑外部子网组时，仅需维护一份网络配置信息，编辑后自动复用至主 \- 备外部子网；  
    * 在网关服务中使用此类外部子网组时，自动在主 \- 备外部子网中分配相同的外部 IP 地址。  
  * 不共用网络配置  
    * 主 \- 备外部子网使用不同的 VLAN 网络、配置不同的 CIDR，但两外部子网中 CIDR 的范围大小、支持的用途以及各用途下的 IP 个数均一致。  
    * 在网关服务中使用此类外部子网组时，需同时分配主 \- 备外部 IP 地址。

### **网关服务**

* NAT 网关、路由网关或浮动 IP；  
  * 允许选择如下外部子网  
    * 属于独立边缘网关；  
    * 属于边缘网关组中的一个边缘网关，且已与另一外部子网关联为主备关系；

    * 浮动 IP；

* L2 网关：需为 L2 网关增加关联边缘网关的字段，允许关联独立边缘网关或边缘网关组。

# 边缘网关

## **创建**

支持创建多个边缘网关，每次手动创建一个，不支持批量创建。

* 基本信息  
  * 名称：一个 Everoute 服务内，不同边缘网关的名称不允许重复。  
  * 描述  
  * 配置类型：增加提示，如需组成边缘网关组，则主、备边缘网关的配置类型需相同。  
  * 虚拟机数量：默认为 2，不允许修改。  
* 网络设置  
  * 所属集群：不同边缘网关可选择相同或不同的所属集群。  
  * 虚拟分布式交换机：不同边缘网关可选择相同或不同的虚拟分布式交换机。  
  * 管理网络/TEP 网络  
    * 不同边缘网关可选择相同或不同的虚拟机网络作为各自的管理网络和 TEP 网络；  
    * 不同边缘网关虚拟机的管理 IP 地址和 TEP 地址均不允许重复。

## **关联**

可选择满足如下要求的两个边缘网关，关联为主备关系，并创建边缘网关组。

* 未关联到任何边缘网关组；  
* 边缘网关中无任何外部子网；  
* 未关联任何 L2 网关；  
* 主备网关的配置类型相同。

关联时，需指定如下内容：

| 字段名 | 说明 | 是否必须 |
| ----- | ----- | :---: |
| 名称 | 填写边缘网关组的名称。一个 Everoute 服务内，不同边缘网关组的名称不允许重复。 | Y |
| 描述 | 填写边缘网关组的描述。 |  |
| 主边缘网关 | 可选当前 Everoute 服务中，满足上述要求的边缘网关。 选项列表中展示：边缘网关名称、~~描述、~~配置类型、所属集群 | Y |
| 备用边缘网关 | 可选当前 Everoute 服务中，满足上述要求，且与主网关配置类型相同的边缘网关。 选项列表中展示：边缘网关名称、~~描述、~~配置类型、所属集群 | Y |

关联边缘网关后，如主备网关属于同一集群，需自动执行如下操作：

* 为边缘网关组创建一个新的放置组规则，要求网关组内的 4 个边缘网关虚拟机优先位于集群中的不同主机。（三节点集群的场景，尽量保证网关虚拟机分散放置）；  
* 自动调度一次，将虚拟机调度至不同物理主机。

* 备用边缘网关为「活动」状态；  
* 主边缘网关的健康状态为「健康」。

## **故障切换** {#故障切换}

1. 此版本中，不支持编辑边缘网关的主备关系，主网关即使当前不处于活动状态，其在主备关系中仍然为主；  
2. 自动切换方式  
   1. 检测到当前的活动网关的健康状态为「异常」后  
      1. 如非活动网关为「健康」状态，则自动将此边缘网关上的网关服务全部切换至非活动网关，并切换活动状态；  
      2. 如非活动网关为「异常」状态，则不触发故障切换，给出报警信息。  
   2. 如当前备用网关为「活动」状态，检测到主网关恢复健康后，不自动切回，仅在提示用户，需用户在确认站点及业务恢复后手动切换。后续版本将考虑增加故障恢复选项，用户手动选择故障恢复后是否自动切回。  
   3. 【下个版本补充支持】如主网关为活动网关，且健康状态为「正常」，备网关为「异常」，则提示用户备用网关为异常状态，请尽快恢复。

| 此版本故障恢复后不会自动切回，原因如下： 如果主集群中的边缘网关虚拟机已恢复， 但业务虚拟机暂未回迁，自动回迁将导致一段时间的流量绕行，且如果集群或边缘网关虚拟机处于不稳定状态（例如维护过程中），可能会反复触发主备切换动作。综合考虑，不支持自动切回，待主站点所有服务恢复后，用户手动回切，对用户来讲可控性更高。 |

### **外部子网 \- 属于独立网关** 

* 名称、描述

### **外部子网组 \- 属于网关组**

* 外部子网组的名称、描述  
  * 无需为主备外部子网分别指定名称和描述信息；  
  * 主备外部子网名称默认为“{{外部子网组名称}} \-1” 和“{{外部子网组名称}} \-2”  
* 所属边缘网关组  
* 是否在主备外部子网中共用网络配置  
  * 共用网络配置  
    * 用户只需指定一次外部子网信息。  
  * 不共用，分别为主备外部子网指定网络配置信息  
    * 外部子网组中，需创建出一个主外部子网和一个备用外部子网；  
    * 其中主边缘网关中的外部子网始终为主，备用边缘网关中的外部子网使用为备用角色；  
    * 用户需分别指定主外部子网和备用外部子网信息。  
* 共享类型  
  * 无论是否在主备边缘网关中共用此外部子网，均指定一次共享类型和所属 VPC；  
  * 不共用的场景下，需对主备外部子网下发相同配置。

如在主备边缘网关中共用网络配置，需指定如下内容：

* 虚拟机网络（VLAN）：选择**主、备用**边缘网关均已关联的虚拟分布式交换机中的一个 access 类型虚拟机网络（VLAN）。  
* CIDR 块  
* 网关  
* 分配地址块

如在主备边缘网关中不共用网络配置，需指定如下内容：

* 主外部子网  
  * 虚拟机网络（VLAN）：选择主边缘网关已关联的虚拟分布式交换机中的 access 类型虚拟机网络（VLAN）。  
  * CIDR 块  
  * 网关  
  * 分配地址块  
* 备用外部子网  
  * 虚拟机网络（VLAN）：选择备用边缘网关已关联的虚拟分布式交换机中的 access 类型虚拟机网络（VLAN）。  
  * CIDR 块  
  * 网关  
  * 分配地址块

### **校验规则**

## **删除**

当外部子网（或外部子网组）关联了以下资源时，不允许删除，必须先解除外部子网与资源间的关联。

# 浮动 IP

## **分配**

需填写如下信息：

* 所属 Everoute 服务  
* 所属 VPC  
* 新增选择「所属边缘网关」字段，需首先选择属于独立边缘网关还是边缘网关组，并手动指定一个边缘网关或边缘网关组。  
* 外部子网，可选择：  
  * 如浮动 IP 属于边缘网关，则需指定此边缘网关中的一个独立外部子网；  
  * 如浮动 IP 属于边缘网关组，则需指定此边缘网关组中的一个外部子网组；

  选择的外部子网或外部子网组，均需要满足如下要求

  * 已关联上述 VPC（或共享类型为「共享型」）；  
  * 已为浮动 IP 分配了地址块。  
* 分配方式  
  * 如浮动 IP 属于独立边缘网关；或属于边缘网关组，但所选外部子网组中只有一个外部子网（即主备边缘网关共用此外部子网）

    则分配方式与 3.0.0 版本相同。

  * 如浮动 IP 属于边缘网关组，且所选外部子网组中包含主、备外部子网。  
    * 自动分配：填写分配 IP 数，任务提交后自动从主、备外部子网中，各分配一个浮动 IP 地址。  
    * 手动分配  
      * 提示：填写外部子网浮动 IP 地址块 {{主 CIDR}} 与 {{备用 CIDR}} 中未被使用的 IP 地址。  
      * 每添加一个浮动 IP，均需要手动填写主 IP 地址和备用 IP 地址。

提交时增加校验：主、备用 IP 地址分别属于主、备用外部子网中的浮动 IP 地址段。

# 对其他功能的影响

## **报警**

增加如下故障及故障转换相关报警：

* 独立边缘网关故障，相关的所有网关服务停止。严重警告  
* 网关组中的活动边缘网关故障  
  * 已自动触发故障转移，相关的网关服务可能发生短暂中断。注意  
  * 无法触发故障转移（非活动网关也处于异常，或其他原因），相关的所有网关服务停止。严重警告  
* 【下个版本补充支持】网关组中的非活动网关故障。注意

## **版本兼容**

对于低于 Everoute 3.2 版本的服务（后文称为低版本服务）。

* 升级前，如配套高版本的 CloudTower 使用，在展示上进行如下处理  
  * 边缘网关、外部子网均在每个 Everoute 服务内单独展示，低版本保持当前展示方式即可。  
  * 浮动 IP、路由网关、NAT 网关、L2 网关列表，均展示当前所有 ER 服务的相关资源，因此对此版本增加的列，例如「所属边缘网关」「备用 IP」等，低版本 ER 服务中的资源在对应列展示为「-」。  
  * 浮动 IP、路由网关、NAT 网关、L2 网关的详情页，低版本保持当前展示方式不变。  
* 升级至 3.2 版本后  
  * 边缘网关  
    * 原边缘网关作为独立边缘网关继续使用；  
    * 如边缘网关上无网关服务 & 外部子网，可以与另一边缘网关关联为网关组。  
  * 外部子网：「所属边缘网关」默认为升级前唯一的边缘网关，不可编辑。  
  * 浮动 IP、路由网关、NAT 网关、L2 网关：「所属边缘网关」默认为升级前唯一的边缘网关，不可编辑。

## **服务异常时的操作限制**

见 [ER 服务异常限制](https://docs.google.com/spreadsheets/d/1lFj0bn2xcjKfFaOr5VBaJCARnZh5mORF7tzubtQHOD4/edit?gid=102252655#gid=102252655)，VPC 页签下的红字部分。🟡 2024-09-12  补充异常描述和限制预期说明

* 支持的操作中，增加如下操作  
  * 创建/编辑/删除边缘网关组  
  * 切换活动网关  
  * 创建/编辑/删除外部子网  
  * 创建/编辑/删除外部子网组  
* 异常状态中，增加边缘网关组相关状态  
  * 创建中、变更中（包括正在变更名称、描述、正在切换活动网关）、删除中  
  * 异常：表示边缘网关组整体状态异常（包括删除任务失败后，可能进入的异常情况）  
  * 边缘网关组中，成员边缘网关的健康状态（即上文[故障切换](#故障切换)中的状态），不包含在表格中边缘网关异常或边缘网关组异常的场景下

上述表格中对应的限制，均在点击对应操作后，通过弹窗提示，无需从界面限制、隐藏操作按钮。
