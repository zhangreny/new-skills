# Everoute 通用 UI Testcase Rules

## 必须遵守

- 顶层分组必须按业务对象或功能域组织，不允许顶层直接按“成功 / 失败 / 报错”分组。
- 第二层优先按页面、动作或功能模块组织，例如列表、详情、创建、编辑、删除、部署、启用、扩容、缩容、关联、解绑。
- 第三层优先按字段、状态、过滤条件、版本差异、权限、异常限制、tab、stepper、section 继续细分。
- `[C]` 具体测试用例只能落在最细分组下，不允许直接挂在大场景或大对象下。
- 同一层内颗粒度必须一致，不允许把“页面分组”和“具体 case”混在同一层。
- 如果某个因素变化后会导致展示、提交流程、校验结果、报错、权限结果、状态变化或联动行为变化，则该因素必须在分组或组合阶段显式体现。
- Step6 只允许输出可复用分组维度，不允许输出 testcase 级结论句。
- Step7 只允许输出组合展开结果，不允许出现 `[C]`。
- Step8 必须显式承接 Step6 的维度和 Step7 的组合，不允许直接重新自由发挥。
- `来源：` 必须精确到文档规则、字段定义、页面区域、控件或报错文案，禁止只写“需求文档”“设计稿”。

## 常见分层模式

- 业务对象 -> 页面/动作 -> 子模块/字段 -> 状态/条件 -> `[C]`
- 功能域 -> 列表/详情/创建/编辑/删除 -> 过滤/校验/展示/限制 -> `[C]`
- 系统服务 -> 版本分支 -> 页面/动作 -> stepper/tab/section -> 字段校验 -> `[C]`
- 业务对象 -> 事件审计与任务 -> 动作类型 -> `[C]`
- 业务对象 -> 权限管理 -> 有权限 / 无权限 -> `[C]`
- 业务对象 -> 异常限制 -> 异常状态 / 禁止操作 -> `[C]`

## 展开规则

- Step6 只保留 6-10 个最有复用价值的核心维度，优先是：场景、入口、对象、联动、版本、环境、状态、容量、许可、结果。
- Step7 组合展开优先使用“场景 × 主变量轴 × 环境/状态 × 结果”的模式，不要把多个独立结果压成一个叶子节点。
- 版本差异、权限差异、异常限制、文案变化、事件审计属于高优先级展开轴，不能被遗漏。
- 同一页面的长名称展示、字段显隐、默认值、禁用态、过滤逻辑等，应优先作为中间分组而不是直接散落为 case。
- 只有当多个值变化后仍然不影响测试结果时，才允许不展开。

## 禁止项

- 禁止在 Step6 输出“某场景下报错 / 成功 / 不报错”的完整结论句。
- 禁止把多个独立变量混成一个维度，例如“许可类型 + 许可方式”。
- 禁止把“正常 / 异常 / 边界”这种抽象词直接当最终分组，不继续展开。
- 禁止在 Step7 提前合并不同版本、不同状态、不同权限或不同结果的组合。
- 禁止在 Step8 把“新集群成功 / 旧集群报错”或“有权限可编辑 / 无权限不可编辑”压成一条 case。
- 禁止来源过粗、层级跳跃、或在长输出后半段明显缩减分组深度。
