# Everoute 通用 UI Testcase Rules

## 生效边界

- 本文件是 `Everoute` 目录内唯一的通用规则入口。
- `Everoute 服务运维 / 分布式防火墙 / 虚拟专有云网络 / 负载均衡` 下的产品主规则文件，与本文件一起构成当前 skill 的唯一生效规则体系。
- 各产品目录下的 `*_product_knowledge.md` 是 Step6-10 的产品知识摘要层，用于对象树、子标题编排习惯、拆分轴、固定块和术语校准。
- `references/Everoute-SmartX-docs` 是产品资料库，只用于 Step6 理解增强、术语校准和资源识别，不直接定义需求范围。
- `Everoute` 目录下的历史需求、历史 testcase、figma 图片和样本目录仍然是原始语料，但只在有明确相似度时作为黄金参考候选。

## 证据优先级

- 当前任务的需求文档、上传 markdown、figma 图片是第一优先级证据。
- 产品知识摘要用于稳定识别对象树、页面动作、固定块和高优先级拆分轴，应先于 SmartX docs 和历史样本读取。
- 本文件和产品主规则文件定义结构、门禁、资源类型、黄金参考与 Step10 细分逻辑。
- SmartX docs 只用于解释对象、对象关系、术语、版本边界、能力边界和资源规则，不能直接把文档里出现过的能力强行纳入测试范围。
- 历史 testcase 样本只在“高相似参考”成立时用于对齐结构、覆盖和数量级，不用于无依据地扩展功能范围。

## Step6-10 契约

- Step6 必须输出 `step6_product_context.md`，说明读取了哪些产品知识摘要、产品资料、历史样本候选、测试资源类型和黄金参考结论。
- Step7 必须输出 `step7_test_blueprint.md`，明确测试对象、subsection、适用维度、资源类型映射、固定块、资源前提、排除项和来源映射。
- Step8 必须输出 `step8_combination_expansion.md`，只做组合展开，不得出现 `[C]`。
- Step9 必须输出 `step9_rule_gate_report.md`、`step9_ui_cases_review.md`、`step9_ui_cases_final.md`。
- Step10 必须输出 `step10_case_split_report.md`、`step10_ui_cases_final.md`，用于继续细分复合 case 并应用数量闸门。
- Step6、Step7、Step8 都是用户确认关卡，只有用户明确表示“继续”后才能进入下一步。
- Step9 在生成最终 testcase 前，应先基于 Step8 叶子路径执行历史标题检索，优先检索已采用黄金参考，再扩展到主产品与补充产品历史样本。

## 通用分层原则

- 顶层优先按业务对象、功能域或稳定的需求主题组织，不允许顶层直接按“成功 / 失败 / 报错”分组。
- 第二层优先按页面、动作或功能模块组织，例如列表、详情、创建、编辑、删除、部署、启用、扩容、缩容、关联、解绑。
- 第三层优先按字段组、状态、过滤条件、版本差异、权限、异常限制、tab、stepper、section、资源类型继续细分。
- 同一层内颗粒度必须一致，不允许把“页面分组”和“具体 case”混在同一层。
- 如果某个因素变化后会导致展示、提交流程、校验结果、报错、权限结果、状态变化或联动行为变化，则该因素必须在分组或组合阶段显式体现。
- `来源：` 必须精确到文档规则、字段定义、页面区域、控件或报错文案，禁止只写“需求文档”“设计稿”。

## 常见分层模式

- 业务对象 -> 页面/动作 -> 子模块/字段 -> 状态/条件 -> `[C]`
- 功能域 -> 列表/详情/创建/编辑/删除 -> 过滤/校验/展示/限制 -> `[C]`
- 系统服务 -> 版本分支 -> 页面/动作 -> stepper/tab/section -> 字段校验 -> `[C]`
- 业务对象 -> 事件审计与任务 -> 动作类型 -> `[C]`
- 业务对象 -> 权限管理 -> 有权限 / 无权限 -> `[C]`
- 业务对象 -> 异常限制 -> 异常状态 / 禁止操作 -> `[C]`

## 通用展开规则

- Step7 中的核心维度优先是：场景、入口、对象、联动、版本、环境、状态、容量、许可、结果。
- Step8 组合展开优先使用“场景 × 主变量轴 × 环境/状态/资源前提 × 结果”的模式，不要把多个独立结果压成一个叶子节点。
- 版本差异、权限差异、异常限制、文案变化、事件审计、资源消耗与归还属于高优先级展开轴，不能被遗漏。
- 同一页面的长名称展示、字段显隐、默认值、禁用态、过滤逻辑、tooltip、删除限制等，应优先作为中间分组而不是直接散落为 case。
- 只有当多个值变化后仍然不影响测试结果时，才允许不展开。
- Step7/8 必须显式承接产品知识摘要中的对象树、固定块和高优先级拆分轴，不允许等到 Step9 临时补猜。

## Step6 资料装载规则

- Step6 先读主产品和补充产品的产品知识摘要，再决定 SmartX docs 与历史样本候选。
- 产品知识摘要用于回答这些问题：对象树怎么组织、子标题通常怎么编排、哪些变量轴必须保留、哪些复合 case 必拆、哪些固定块不能漏。
- SmartX docs 的目标是挑出“有助于理解当前材料”的产品资料，不是把整套产品文档都读一遍。
- 默认选择顺序：
  1. 主产品和补充产品的产品知识摘要
  2. 同产品管理指南
  3. 同产品技术白皮书
  4. 术语表
  5. 发布说明
  6. 安装与升级指南
- 有助于理解的判定标准：
  - 能解释需求文档中的对象、对象关系、网络模式、主备关系或术语
  - 能帮助识别 figma 图片中的页面入口、字段组、网络配置块、对象层级
  - 能补齐历史 testcase 中反复出现但需求文档写得不清的前置条件、配置步骤或能力边界
  - 能解释版本差异、发布变更、启用条件、部署前提
- 如果材料出现版本、发布或兼容差异，Step6 必须考虑发布说明。
- 如果材料出现部署、启用、安装、升级或迁移语义，Step6 必须考虑安装与升级指南。

## 黄金参考选择规则

- 黄金参考不是默认路径，只有在当前任务与某份历史样本高度相似时才启用。
- 选择黄金参考时至少要满足：
  - 产品组合相似
  - 主对象相似
  - subsection 相似
  - 关键维度轴相似
  - 关键资源类型相似
- Step6 必须显式写出：
  - `已采用黄金参考：...`
  - 或 `未采用黄金参考：...`
- 若未采用黄金参考，Step9 报告中必须写“跳过黄金参考对比”。

## 数量规则

- Step9 仅在采用黄金参考时做相对数量级检查，默认下限为 `0.8x`，默认上限 `1.2x` 只作为提示，不作为强制压缩门禁。
- Step10 负责 Figma 数量闸门。若当前 workdir 存在 Figma 下载目录，则以 `ceil(figma_total_bytes / 12000)` 作为最低数量目标。
- 若当前 workdir 没有 Figma 下载目录，则 Step10 跳过数量闸门，但仍必须执行结构化细分审计。
- 数量永远不是第一判定标准。对象树、subsection、关键维度轴、固定块和资源类型覆盖优先于数量。

## Step9 标题语义规则

- Step9 的 `[C]` 标题应优先写成“动作 + 条件 + 结果”的具体交互语义，而不是抽象结果桶。
- 历史样本中的 case 标题可作为语义模板，但不得直接照搬复合 case 标题。
- `显隐 / 默认值 / 过滤 / 禁用 / 报错 / 提交成功 / 文案变化` 是结果轴，不是固定的最终叶子标题模板。
- 生成前应优先根据 `主对象 + subsection + 页面动作 + 网络类型 + 通信方向 + 主备关系 + 结果类型` 检索历史近邻标题。

## Step10 细分规则

- Step10 必须先运行 gate 脚本，再决定是否继续细分。
- 若 gate 脚本返回 `needs_more_split=true`，则继续细分，禁止把“达到某轮次”当成停止依据写进生成指令。
- Step10 的强制拆分信号至少包括：
  - case 名含 `同时 / 且 / 并且 / 反复 / 复合 / 批量 / 多个`
  - 单条 case 混合多个动作、多个结果、多个模式、多个对象或多个独立资源
  - case 所在父节点缺失产品知识摘要定义的高优先级维度轴
  - case 同时覆盖 `展示+校验`、`编辑+结果`、`成功+失败`、`主+备`、`客户端+服务端` 等独立结果
- 历史样本中存在的复合 case 只可作为“需要继续拆”的信号，不得直接继承为单条 `[C]`。

## 通用测试资源类型总表

### 版本资源

- 产品版本
- CloudTower 版本
- 功能版本分支

### 环境资源

- 集群
- 节点
- 站点/机房/双活环境
- 主环境 / 旧环境 / 新环境

### 对象资源

- 业务对象
- 页面对象
- 卡片、tab、stepper、section

### 网络资源

- 网络类型
- 子网
- 网络组
- IP 池 / VIP / LIP / TEP IP
- 通信方向
- 主备关系

### 关联资源

- 关联对象
- 被引用对象
- 删除限制对象
- 过滤范围对象

### 权限资源

- 角色
- 权限开关

### 审计资源

- 事件审计
- 审计动作类型

- 报错文案
- 说明文案

- 过期状态
- 消耗动作
- 归还动作

## 通用禁止项

- 禁止在 Step6 输出 testcase、组合树或 testcase 级结论。
- 禁止把多个独立变量混成一个维度，例如“许可类型 + 许可方式”。
- 禁止把“正常 / 异常 / 边界”这种抽象词直接当最终分组，不继续展开。
- 禁止在 Step8 提前合并不同版本、不同状态、不同权限、不同资源模式或不同结果的组合。
- 禁止在 Step9 把“显隐检查 / 字段校验 / 提交结果 / 报错结果 / 权限结果”压成一条 case。
- 禁止在 Step9 把 `显示入口 / 显示默认值 / 显示说明文案 / 显示资源范围 / 列表可见 / 详情可见 / 状态正确 / 提交成功 / 按钮置灰 / 提示必填 / 提示格式错误 / 提示资源冲突 / 保留原状态 / 详情显示限制` 直接作为最终 `[C]` 标题。
- 禁止在 Step10 保留明显的复合 case，而只做表面数量补齐。
- 禁止来源过粗、层级跳跃，或在长输出后半段明显缩减分组深度。
