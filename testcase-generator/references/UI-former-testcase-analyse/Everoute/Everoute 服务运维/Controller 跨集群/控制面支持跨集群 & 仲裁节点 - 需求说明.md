# 概述

## **背景**

目前 Everoute Controller 集群中的所有虚拟机都需要运行在同一个 SMTX 集群内，为多个 SMTX 集群提供网络与安全服务。在此情况下，如果 ER 服务所在的集群整体异常，将导致故障期内无法对其他被管集群中的安全策略进行管理操作。如果所在集群无法恢复，相关的安全策略等内容也均无法复原，可能需要重新部署一个 Everoute 并对已下发的策略进行覆盖或清空处理，严重影响其他集群的正常使用。

此外，如果 Everoute 所在集群需进行改造、搬迁、重建，目前无法对 Everoute 服务做迁移操作，只能采用重建的方式。

在佳通轮胎的使用案例中，为了将集群异常的影响范围仅控制在当前集群内，客户为每个集群单独部署了一个 Everoute 服务，此方式严重增加了管理成本，无法发挥安全策略跨集群的能力。由于业务分布在多个集群中，但安全策略无法跨 Everoute 服务，因此客户需要在每个 ER 服务里单独创建一整套安全策略，非常不便。

因此，希望 Everoute Controller 可以提供跨集群的能力，主要为如下：

1. 一个 Everoute 中的多个 Controller 虚拟机，支持部署在不同 SMTX 集群内。  
2. 提供仲裁机制，如已存在如下任意一类仲裁节点，Everoute Controller 集群可以利用仲裁节点的能力，在故障时参与选中，保证正常进行故障切换。  
   1. SMTX 双活集群的仲裁节点  
   2. CloudTower HA 的仲裁节点（详见[CloudTower 高可用 - 需求说明](https://docs.google.com/document/d/1k-eAQemED-S8ev-d0H7V6UL7tHwD612SrZSqNlA92a0/edit?tab=t.0#heading=h.wfhi0u4x2tzo)、[CloudTower 高可用技术方案](https://docs.google.com/document/d/1pwdyrV3m4tez6mIFCT9S1eJNvWmWs0CryritA2-PXuk/edit?tab=t.0#heading=h.ckxcg2h9usmo) ）

   

3. 提供跨集群迁移能力，包括如下  
   1. 节点替换：任一 Controller 虚拟机迁移/重建到新的 SMTX 集群（迁移过程中无中断，要求迁移后仍与其他 Controller 节点互通）。  
   2. 服务迁移：Controller 集群整体迁移/重建到新的 SMTX 集群（迁移过程中控制面中断）。

各种情况下的使用场景和部署建议详见[使用场景]() Tab。

## **基础设定**

1. 将 Everoute Controller 区分为仲裁节点和非仲裁节点（后文称为工作节点），不同节点的作用、配置项以及部署方式均不同。  
2. 部署时，增加部署方式选项，可选择  
   1. 单集群部署（非双活集群）：所有 Controller 位于同一集群，与当前部署方式相同。  
   2. 双活集群部署：所有 Controller 位于同一个 SMTX 双活集群，在双活仲裁节点部署 Controller 仲裁，其他 Controller 节点平均分布在两可用域。  
   3. 跨集群部署：所有 Controller 位于不同集群，可选择是否配置仲裁节点（存在 CloudTower HA 仲裁节点时可选）。  
3. 跨集群部署方式下  
   1. 支持将 Controller 节点（包括仲裁节点）部署在不同 SMTX 集群中，由于不同集群通常不在同一个二层网络内，因此需为每个 Controller 节点单独配置 IP、子网掩码、网关信息，保证节点间三层连通。

   2. 使用仲裁节点的前提是，客户已经在双活集群或 CloudTower HA 的部署过程中完成至少一个仲裁节点的部署，Everoute 仅在已有的仲裁节点中部署相关进程，Everoute 自身不支持部署新的仲裁节点。

| 说明：由于 Everoute 依赖 CloudTower 作为管理面，如存在仲裁需求，应为 CloudTower 和 Everoute 同时配置仲裁，否则如果 CloudTower 异常或无法恢复，Everoute Controller 自身的高可用或仲裁机制也无法发挥作用。 |
| :---- |

4. 提供跨集群迁移能力  
   1. 节点替换  
      * 支持删除任一 Controller 虚拟机，即原虚拟机弃用，并在指定集群创建新的 Controller 虚拟机；  
      * 需保证替换期间服务整体可用，且新的 Controller 虚拟机仍需要保持与其他 Controller 虚拟机连通。  
   2. ~~服务迁移~~ 🟡 2025-04-18，此版本暂不提供，[讨论](https://smartx1.slack.com/archives/C014SKH8823/p1744964971225559?thread_ts=1744698692.747619&cid=C014SKH8823)  
      * ~~支持将 Everoute Controller 集群整体迁移至新的 SMTX 集群；~~  
      * ~~迁移过程中控制面会有中断，迁移完成后，需保证所有新的 Controller 节点间相互连通。~~  
5. 由于节点替换和迁移的能力，和切换部署模式的场景有重合，为尽量提供灵活的部署方式，应对客户场景下不同的变更需求，考虑在部署完成后不单独提供「转换部署模式」操作，而是通过节点替换和服务迁移达到转换目的。

   典型场景如下：

| 原部署模式 | 预期变更 | 变更后的部署模式 | ER Controller 处理方式 |
| :---: | ----- | :---: | ----- |
| 单集群部署 （非双活集群） | SMTX 集群搬迁/重建 | 单集群部署 （非双活集群） | 服务迁移，整体迁移至新的 SMTX 集群。 |
|  | SMTX 集群转双活 | 双活集群部署 | 集群转双活后，ER Controller 均位于优先可用域； 节点替换 一个 Controller 替换为仲裁节点 剩余半数 Controller 替换为次级可用域节点 |
|  | 已有双活集群，目前 Controller 均部署在优先可用域，转为仲裁部署方式 |  |  |
|  | 已有 ER 服务，扩展至跨集群部署 | 跨集群部署 | 节点替换，多个 Controller 分别替换为其他 SMTX 集群中的节点。 |
| 双活集群部署 | SMTX 集群搬迁/重建 | 双活集群部署 | 服务迁移，整体迁移至新的双活集群。 |
| 跨集群部署 | SMTX 集群搬迁/重建 | 跨集群部署 | 节点替换，多个 Controller 分别替换为其他 SMTX 集群中的节点。 |
|  | 其中某 SMTX 集群转双活 | 双活集群部署 | 节点替换 一个 Controller 替换为双活集群的仲裁节点 剩余半数 Controller 替换为次级可用域节点 |

校验规则：不同 Controller 之间

* 所属集群、虚拟机网络、子网掩码、网关信息允许重复；  
* 所有管理 IP 地址之间不允许重复；  
* 所有管理 IP 地址都不允许和网关地址重复；  
* （3.4.0 之前版本）管理虚拟 IP 地址不允许和所有其他管理 IP 地址、网关地址重复。

# 展示

展示 Everoute Controller 虚拟机详情的位置：

* 需对仲裁节点单独标记，表明其类型为「仲裁节点」。  
* 如仲裁节点为 SMTX 集群中的虚拟机，其则其信息展示方式与工作节点相同。  
* 如仲裁节点为双活集群的仲裁节点  
  * 无「所属主机」、「虚拟机网络」字段；  
  * CPU、存储容量及内存使用率字段，需单独从双活集群监控信息中获取。  
* 删除「管理虚拟 IP」字段。

# 扩容

在「服务运维」的「Everoute Controller」部分，可增加 Everoute controller 虚拟机数量，增加数量时：

* 如当前不存在仲裁节点，可选择新节点是否为仲裁节点。  
* 如当前已存在仲裁节点，则新节点默认为工作节点，不允许选择。  
* 对于新增加的工作节点，需要指定所属集群、所属可用域（所属集群为双活集群时，需选择此项）、虚拟机网络、IP 地址、子网掩码、网关信息。  
* 对于新增加的仲裁节点，需选择双活仲裁或 CloudTower 仲裁节点。  
* 提示：新节点的管理 IP 地址需与其他 Everoute Controller 之间的管理 IP 地址需相互连通。

# 节点替换

对于每个 Everoute Controller，均支持替换为新节点，即删除此节点，并创建新的 Controller 节点，创建时需指定新节点的如下信息：

* 前提条件：当前 Everoute Controller 数量为 3 或 5（单节点场景下不支持替换）🟡 2025-09-03 补充限制，[Slack](https://smartx1.slack.com/archives/C014SKH8823/p1756868182150469?thread_ts=1755511964.235139&cid=C014SKH8823)  
* 是否为仲裁节点  
  * 如其他 Controller 中不存在仲裁节点，可选择新节点是否为仲裁节点。  
  * 如其他 Controller 中已存在仲裁节点，则新节点默认为工作节点，不允许选择。  
* 对于新增加的工作节点，需要指定所属集群、所属可用域（所属集群为双活集群时，需选择此项）、虚拟机网络、IP 地址、子网掩码、网关信息。对于新增加的仲裁节点，需选择双活仲裁或 CloudTower 仲裁节点。  
* 提示：新节点的管理 IP 地址需与其他 Everoute Controller 之间的管理 IP 地址需相互连通。

替换节点的操作中，替换后节点默认与原节点的上述信息相同，即不变更所属集群，仅在当前集群中 Controller 故障时替换为一个新的 Controller 使用。如需将新节点替换至其他 SMTX 集群中，用户可按需修改上述信息，编辑「所属集群」后，对应其他的虚拟机网络、IP 等信息自动清空，用户可手动重新输入。🟡 2025-03-07 根据需求评审补充

# 节点分布要求

## **集群分布要求**

1. 无仲裁节点时  
   1. 可将所有 Everoute Controller 部署在同一个 SMTX 集群（或双活集群可用域），与已有版本的方式相同，提供节点级别高可用。  
   2. 可分布在 3 个及以上 SMTX 集群（或双活集群可用域），提供集群/可用域级别高可用。  
   3. 非常不建议部署在 2 个 SMTX 集群（或双活集群可用域）中，此时虽为跨集群部署，但 Everoute Controller 在多个集群中的分布不均衡，一旦多数 Controller 节点所在的集群/可用域故障，将导致 Everoute 服务不可用，实际无法提供集群级别的高可用。  
2. 有仲裁节点时  
   1. 可将仲裁节点及其他工作节点部署于 3 个及以上 SMTX 集群（或双活集群可用域），提供集群/可用域级别高可用。  
   2. 非常不建议将所有节点部署于同 1 个或 2 个 SMTX 集群（或双活集群可用域）中，此时仍存在 1-c 中的问题，一旦多数 Controller 节点所在的集群/可用域故障，将导致 Everoute 服务不可用。

存在上述 1-c 或 2-b 的情况时，需在在「服务运维」的「Everoute Controller」页面给出提示。

| 说明：可能造成上述问题的操作是 部署 Everoute 服务时，选择跨集群部署，并将多个工作节点/仲裁节点选择为相同集群。 扩容/替换节点/迁移时，选择的新节点与已有的工作节点/仲裁节点位于相同集群。 在上述操作期间，不做严格的检查和限制，仍允许部署/扩容/替换/迁移，仅在 Controller 展示的位置给出提示。以防存在特殊情况，如另一集群尚未准备完成，暂时使用两集群的场景下，可作为过渡方案。 |
| :---- |

提示场景，如下多条之间为“且”的关系

* Controller 部署在多个集群，或部署在双活集群的多个可用域。  
* 任一非双活集群或双活集群的任一可用域中，包含的 Controller 数量超过了总数的一半。

（其中，双活集群特指 6.0.0 及以上版本集群，且 Everoute 服务为 3.4.0 及以上版本的场景，其他场景均作为非双活集群处理）

具体判断如下：[Slack](https://smartx1.slack.com/archives/C014SKH8823/p1757493478821029?thread_ts=1757387156.785409&cid=C014SKH8823)

* ER 3.4 以下版本：不判断，无提示  
* ER 3.4 及以上版本  
  * 所有 Controller 部署在 1 个集群，要求如下，不满足则提示  
    * 600 及以上双活集群的单个可用域内，Controller 数量不能超过总数的一半  
  * 所有 Controller 部署在多集群，要求如下（且的关系），不满足则提示：  
    * 单个非双活集群或 600 以下双活集群中，Controller 数量不能超过总数的一半  
    * 600 及以上双活集群的单个可用域内，Controller 数量不能超过总数的一半

提示内容：

* 多数节点位于同一个非双活集群时：当前 {数量} 个 Everoute Controller 部署在集群 {名称}  中，无法提供集群级别高可用，此集群异常时，Everoute 服务将无法使用。  
* 多数节点位于同一个双活集群可用域时：当前 {数量} 个 Everoute Controller 部署在集群 {名称}  的 {优先可用域/次级可用域} 中，无法提供集群/可用域级别高可用，可用域异常时，Everoute 服务将无法使用。

## **放置组规则**

在进行部署、扩容、节点替换、迁移时，如存在如下情况，需自动创建放置组规则。

* 多个 Controller 位于不同主机  
  * 无仲裁节点时：如任意两个工作节点位于同一集群或同一可用域，需要创建优先放置组规则，将多个 Controller 优先放置于不同物理节点。  
  * 有仲裁节点时  
    * 如仲裁节点为 CloudTower HA 的仲裁节点（并且是集群内 vm），此时任意两个节点（包括工作节点和仲裁节点）位于同一集群或同一可用域，需要创建优先放置组规则，将多个 Controller 优先放置于不同物理节点。  
    * 如仲裁节点为双活集群的仲裁节点，由于仲裁节点不属于任一可用域，此时不会出现仲裁节点与工作节点位于同一可用域的情况，因此只考虑工作节点。任意工作节点位于同一集群，需要创建优先放置组规则，将多个 Controller 优先放置于不同物理节点。  
* Controller 位于双活集群的特定可用域 🟡 2025-03-07 根据需求评审补充  
  * Controller 部署在双活集群时，希望只在当前可用域内做 HA，可用域整体故障时，无需 HA 到次级可用域，而是由次级可用域中的其他 Controller 提供服务，因此需要增加放置组规则。  
  * 当前版本：创建放置组规则，选择当前可用域中的所有主机，要求 Controller 必须放置在对应主机上。  
  * 6.3.0 及更高版本：创建放置组规则，Controller 必须放置在当前可用域内主机上（集群扩容时 ELF 动态调整可用域的主机列表，[讨论](https://smartx1.slack.com/archives/C18JK9VV3/p1741327379802229)）。

# 增加 Reconcile 操作

🟡2025-07-09 增加，[Slack 讨论](https://smartx1.slack.com/archives/C014SKH8823/p1751867994758769?thread_ts=1751866421.914779&cid=C014SKH8823)

## **背景**

目前在执行 Everoute 运维管理的相关操作时，可能会出现部分步骤成功而其他步骤失败的情况，这时用户重新在此执行编辑操作，但因为界面信息已经显示为更新后的状态，用户只能提交一个内容完全相同的表单来重新启动编辑任务，从而重试之前失败的步骤。

例如，编辑 Everoute Controller 的管理 IP 地址时，后端需要执行两个步骤：1. 编辑 Everoute Controller 的 IP 地址；2. 把新的 IP 地址同步到 VPC 边缘网关、LB 实例、以及 VPC 已关联的集群上。此时如果 VPC 或 LB 自身的状态异常，可能导致步骤 1 成功、步骤 2 失败，但从界面查看，Controller 的 IP 地址已经修改完成。

目前已经允许重新提交表单的操作有：Everoute 服务升级、Everoute Controller 扩容、LB 实例扩容。但这种方式不仅用户体验较差，还要求前端针对特定表单进行专门的处理。

因此希望单独支持一个用户操作，可在点击后进行前后端的状态同步和恢复，具体涉及如下资源：

* Everoute 服务  
* LB 实例  
* VPC 边缘网关  
* VPC 已关联集群

# 对其他功能的影响

## **升级为双活部署**

🟡 2025-09-09 补充，[Slack](https://smartx1.slack.com/archives/C014SKH8823/p1757389860795759?thread_ts=1757387156.785409&cid=C014SKH8823)

从此版本开始，支持将 Controller 部署在双活集群中（后简称为“双活部署”），此部署方式需同时满足如下两个条件：

1. Everoute 服务为 3.4.0 及以上版本  
2. 所属的 SMTX OS（ELF）双活集群为 6.0.0 及以上版本

如上述任一条件不符合，均将双活集群作为非双活集群处理，即：

* 不体现 Controller 节点所属的可用域，仅体现其属于此集群（创建时预期都在优先可用域中，后续由于 HA、迁移等动作，可能被调度至次级可用域）。  
* 不支持复用双活集群的仲裁节点。

此时在编辑、扩容、提升配置、替换操作中，均与单集群操作相同。

后续在进行如下操作后，可能将普通部署模式转为双活部署模式：

1. 升级 Everoute 服务，从低版本升级至 3.4.0 及以上版本  
2. 升级 SMTX OS（ELF）双活集群，从 5.x 版本升级至 6.0.0 及以上版本  
3. 对 6.x 的 SMTX OS（ELF）集群进行转换操作，从非双活集群转换为双活集群

发生上述操作后，会存在一个中间状态，此时虽然已满足双活部署的条件，但 Controller 节点仍属于整个集群，没有所属可用域的属性（spec 中的可数可用域为空），也没有相关的放置组规则限制其必须属于某一可用域，其所在的可用域仍然可能随后续的各种调度操作发生变化，且两可用域中的工作节点数量不能保证均分。

工作流程上，此时需要用户手动对每个 Controller 节点逐个执行替换动作，将其划分至特定可用域，执行后，后端默认创建放置组规则，此时才转为了双活部署模式。

* 查看 Everoute Controller 节点：按照 Controller 当前所在的可用域（即 Status 中记录的可用域信息）展示，不考虑 spec 中的可用域信息是否为空

    * 提交编辑后，将 status 中的所属可用域传至 spec 中，此时后端根据 spec 值自动创建相关的放置组规则  
    * 「增加 Everoute Controller 数量」弹窗提交时，仍检查编辑后是否满足 3 节点或 5 节点要求

## **端口占用**

由于多个 Everoute 服务可复用相同的仲裁节点，复用时，每个 Everoute 服务需单独占用仲裁节点中的一个端口。因此需对仲裁节点增加端口占用说明，可预给定一个端口范围，要求用户对此范围内的端口开通防火墙规则，每个 ER 服务部署时，按需使用即可。

# 跨集群部署

## **非双活集群**

### **部署方式**

1. #### **跨 3 个及以上集群部署**

   如客户拥有 3 个及以上 SMTX 集群，可以在多个集群中分别部署 Everoute Controller 虚拟机。

* **3 个 Controller 节点**

  可以分别部署在 3 个 SMTX 集群中（即 1+1+1 模式），当其中一个集群故障时，仍剩余两个可用 Controller 节点，ER 服务整体仍继续运行

  ![][image1]

* **5 个 Controller 节点**

  可以部署在 3-5 个 SMTX 集群中，当其中 1-2 个集群故障时，仍剩余 3 个可用 Controller 节点，ER 服务整体仍继续运行

  此场景下，强烈建议将 Everoute Controller 部署在三个及以上 SMTX 集群中。如果仅分布在两个集群，即对于 3 节点的 ER Controller 集群，其中一个 Controller 位于集群 A，另外两个位于集群 B，此时一旦集群 B 故障，对 Everoute Controller 而言多数节点故障，ER 服务将无法运行，无法达到跨集群高可用的效果。

2. #### **跨 2 个集群 \+ 第三地仲裁节点**

   在同城双活数据中心的架构下，用户可能不使用 SMTX 双活集群，而是在两机房分别部署一个非双活集群，同时运行业务，两机房之间互为主备，同时提供一个独立的第三地部署仲裁节点。

   此时可以同时使用 CloudTower HA 和 Everoute Controller 高可用，复用相同的仲裁节点，保证两集群之间的高可用。

## **双活集群**

配合 SMTX 双活集群使用，在多个可用域内部署 Everoute Controller 虚拟机，同时在双活集群的仲裁节点中部署 ER Controller 的仲裁节点，故障时参与选主。可在两个可用域内部署相同数量的 Controller 节点（图中为 2+2+1 的部署模式，也可以采用 1+1+1 的方式），优先可用域故障时，由于仲裁节点参与选主，此时服务仍有多数节点存活，将继续通过次级可用域内的 Controller 提供服务。

![][image18]

# 计划内的集群搬迁/弃用

如 ER Controller 所在的 SMTX 集群由于架构、机房调整等将被弃用或重建，可对其中的 Controller 进行计划内的替换或整体迁移。

1. 跨集群部署时，以场景一种的部署方式为例，如将其中 Controller 所在的 SMTX 集群将弃用，可替换其中的 ER Controller 节点，在新集群重建一个新的 Controller 节点继续使用。

![][image19]

2. 集群内部署时，如 Everoute Controller 整体所在的集群将弃用，可通过服务迁移的操作，将整体服务均迁移至新集群。
