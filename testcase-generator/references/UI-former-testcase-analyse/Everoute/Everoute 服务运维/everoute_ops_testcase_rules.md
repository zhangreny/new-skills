# Everoute 服务运维 UI Testcase Rules

## 必须遵守

- Step6 先读取 `everoute_ops_product_knowledge.md`，以版本、集群、节点、许可和固定块为骨架，再决定安装与升级指南、发布说明和历史样本的装载方式。
- 顶层优先按“部署 ER 服务 / ER 设置 / 事件审计与任务 / 权限管理 / 异常操作限制”组织。
- 与 Controller、系统服务、仲裁节点、集群扩缩容、许可更新相关的需求，必须显式保留版本分支。
- 只要 ER 版本不同导致页面结构、字段、按钮、tip、节点模型或能力边界不同，就必须先拆版本层，再落后续分组。
- 只要集群类型不同导致页面结构或校验不同，就必须单独拆出普通集群、双活集群、跨集群、仲裁节点等分组。
- 工作节点、仲裁节点、所属集群、节点数量、批量填充、提交校验、替换节点等必须保留为独立子模块，不允许合并成“配置校验”大杂烩。
- 许可相关场景在运维产品中通常不是唯一主轴，必须同时保留部署形态、版本差异、节点/集群关系这几个主变量轴。

## 常见分层模式

- 部署 ER 服务 -> ER 版本分支 -> section/stepper -> 字段或校验点 -> `[C]`
- ER 设置 -> 详情页 / 节点替换 / 增加节点数量 -> 子模块 -> 状态/条件 -> `[C]`
- 功能主题 -> 集群类型 / 节点类型 / 版本分支 -> 字段或过滤逻辑 -> `[C]`
- 事件审计与任务 -> 动作类型 -> `[C]`
- 权限管理 -> 有权限 / 无权限 -> `[C]`
- 异常操作限制 -> 异常状态 -> 可操作 / 不可操作对象 -> `[C]`

## 展开规则

- 高优先级展开轴：ER 版本、Tower 版本、集群类型、节点类型、部署形态、权限、异常状态。
- 如果一个动作同时受“版本差异 + 集群类型 + 节点类型”影响，优先按版本，再按集群/节点继续分层。
- 集群或节点的过滤逻辑、灰色禁用原因、tip 文案、按钮显隐、默认值和提交校验，需要独立展开，不要只写一个“字段校验正确”。
- 与许可相关的运维场景，优先使用“场景 / 关联功能 / 许可子类 / 环境对象 / 容量状态 / 结果”做组合。
- 对于同一页面上反复出现的字段展示和校验项，应优先合并到字段级分组，再在字段下展开状态和结果。
- Step7/8 必须把历史样本中的“复合点击”“批量填充”“多个工作节点”“多个集群”视为继续拆分信号，而不是继承为单条 case。

## 测试资源类型

### 稳定对象与环境资源

- ER 版本
- Tower 版本
- 集群类型
- 节点类型
- 工作节点 / 仲裁节点
- 所属集群
- 节点数量

### 运维许可资源

- 许可对象：集群、节点、关联功能
- 许可方式：按集群、按节点、按子功能资源
- 额度状态：充足 / 刚好 / 不足 / 已满
- 结果资源：可部署 / 不可部署 / 可扩容 / 不可扩容 / 文案提醒

### 权限与审计资源

- 权限角色
- 审计动作
- 异常状态

## 资源展开提示

- 涉及部署、启用、扩缩容、许可更新时，优先把“版本差异 + 集群关系 + 节点关系 + 许可资源”一起保留。
- 如果许可不是唯一主轴，也不能丢掉部署形态、节点关系和版本分支。

## 禁止项

- 禁止把 ER 3.4 及以上与低于 3.4 的差异压成同一组。
- 禁止把普通集群、5.x 双活、6.x 双活等不同部署形态混在同一个字段分组下。
- 禁止把“字段必填 + 格式校验 + 过滤逻辑 + 提交结果”压成一条 case。
- 禁止忽略事件审计、权限管理、异常操作限制这三类运维高频固定块。
- 禁止在许可类运维需求里只看许可，不看部署版本、集群关系和节点关系。
- 禁止把样本中的“复合点击”或“批量填充后继续操作”原样落为单个 `[C]`。
