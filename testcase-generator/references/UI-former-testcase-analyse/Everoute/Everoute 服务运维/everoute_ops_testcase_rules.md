# Everoute 服务运维 UI Testcase Rules

## 必须遵守

- Step6 先读取 `everoute_ops_product_knowledge.md`，以版本、集群、节点、许可和固定块为骨架，再决定安装与升级指南、发布说明和历史样本的装载方式。
- 顶层优先按“部署 ER 服务 / ER 设置 / 事件审计与任务 / 权限管理 / 异常操作限制”组织。
- 若需求属于系统服务 IP / 网络编辑，顶层优先按 `ER 概览提示 / 虚拟管理 IP / ControllerIP / LBVMIP / GatewayVM / VPC 集群关联 / TEP IP 池 / 事件审计与任务 / 权限管理 / 异常操作限制` 组织，而不是笼统并入 `ER 设置`。
- 与 Controller、系统服务、仲裁节点、集群扩缩容、许可更新相关的需求，必须显式保留版本分支。
- 只要 ER 版本不同导致页面结构、字段、按钮、tip、节点模型或能力边界不同，就必须先拆版本层，再落后续分组。
- 只要集群类型不同导致页面结构或校验不同，就必须单独拆出普通集群、双活集群、跨集群、仲裁节点等分组。
- 工作节点、仲裁节点、所属集群、节点数量、批量填充、提交校验、替换节点等必须保留为独立子模块，不允许合并成“配置校验”大杂烩。
- 系统服务网络编辑需求至少保留 `ER < 3.2.0`、`3.2.0 <= ER < 3.4.0`、`ER = 3.4.0` 三段版本边界；`ER = 3.4.0` 的特殊能力边界不能并入泛化的 `ER >= 3.2.0`。
- 许可相关场景在运维产品中通常不是唯一主轴，必须同时保留部署形态、版本差异、节点/集群关系这几个主变量轴。
- `未配置 TEP IP 提示` 与 `信息类告警` 属于系统服务网络编辑需求的默认固定块，不能因为它们不是主弹窗字段而省略。

## 常见分层模式

- 部署 ER 服务 -> ER 版本分支 -> section/stepper -> 字段或校验点 -> `[C]`
- ER 设置 -> 详情页 / 节点替换 / 增加节点数量 -> 子模块 -> 状态/条件 -> `[C]`
- 功能主题 -> 集群类型 / 节点类型 / 版本分支 -> 字段或过滤逻辑 -> `[C]`
- 事件审计与任务 -> 动作类型 -> `[C]`
- 权限管理 -> 有权限 / 无权限 -> `[C]`
- 异常操作限制 -> 异常状态 -> 可操作 / 不可操作对象 -> `[C]`

## 系统服务 IP 调整专项规则

- 顶层对象默认按 `ER 概览提示 / 虚拟管理 IP / ControllerIP / LBVMIP / GatewayVM / VPC 集群关联 / TEP IP 池 / 固定块` 组织；只有材料明确缺失某对象时才允许省略对应子树。
- `LBVMIP` 子树继续继承 LB 规则，`GatewayVM / VPC 集群关联 / TEP IP 池` 子树继续继承 VPC 规则；运维主规则负责版本骨架、固定块和跨对象总装结构，禁止把这些子树扁平成一个“系统服务网络设置”分组。
- 版本轴至少保留 `ER < 3.2.0`、`3.2.0 <= ER < 3.4.0`、`ER = 3.4.0`；只要页面结构、字段、可编辑性或二次确认内容不同，就必须单独展开。
- `管理网络` 与 `TEP 网络` 必须分开，不能把两个网络域压成一个“网络配置”分组。
- 编辑行为至少展开：有无编辑入口、输入框/文本态/不可编辑态、批量填充、留空自动分配、切换资源后清空已有值、实质修改才触发二次确认。
- 二次确认弹窗必须继续拆出：顶部提示是否出现、管理网络修改分区、TEP 网络修改分区、每个 VM/主机的修改项是否按需显示、自动分配场景是否单独出现。
- 旧版本扩容场景必须单独保留：只允许为新增节点填写 TEP IP、既有节点不可编辑、批量填充不能误覆盖既有节点。
- `未配置 TEP IP 提示` 不能只按普通字段校验处理，至少要保留概览页注意提示、异常数字统计、点击后的分类型主机清单，并把 `ER 代理异常` 与 `未配置 TEP IP` 两类异常分开展开。
- `信息类告警` 不能省略，至少保留 `持续 1h` 触发时机、告警文案、触发原因、影响、解决方法和告警等级。

## 展开规则

- 高优先级展开轴：ER 版本、Tower 版本、集群类型、节点类型、部署形态、权限、异常状态。
- 如果一个动作同时受“版本差异 + 集群类型 + 节点类型”影响，优先按版本，再按集群/节点继续分层。
- 集群或节点的过滤逻辑、灰色禁用原因、tip 文案、按钮显隐、默认值和提交校验，需要独立展开，不要只写一个“字段校验正确”。
- 与许可相关的运维场景，优先使用“场景 / 关联功能 / 许可子类 / 环境对象 / 容量状态 / 结果”做组合。
- 对于同一页面上反复出现的字段展示和校验项，应优先合并到字段级分组，再在字段下展开状态和结果。
- 系统服务网络编辑场景下，旧版本扩容节点、主机名称排序、自动分配顺序、每 VM / 每主机 IP 和 VDS 关联关系都应作为可展开资源，不要隐藏在描述句里。
- Step7/8 必须把历史样本中的“复合点击”“批量填充”“多个工作节点”“多个集群”视为继续拆分信号，而不是继承为单条 case。

## 测试资源类型

### 稳定对象与环境资源

- ER 版本
- Tower 版本
- 集群类型
- 节点类型
- 工作节点 / 仲裁节点
- 所属集群
- 节点数量

### 系统服务网络资源

- 虚拟机网络
- 管理 IP / 管理 VIP
- 子网掩码
- 网关
- VDS
- TEP IP 池
- 每 VM / 每主机 IP
- 主机名称排序
- 自动分配 / 手动填写模式
- 新增节点 / 既有节点
- 代理异常 / 未配置 TEP IP 异常类型

### 运维许可资源

- 许可对象：集群、节点、关联功能
- 许可方式：按集群、按节点、按子功能资源
- 额度状态：充足 / 刚好 / 不足 / 已满
- 结果资源：可部署 / 不可部署 / 可扩容 / 不可扩容 / 文案提醒

### 权限与审计资源

- 权限角色
- 审计动作
- 异常状态

## 资源展开提示

- 涉及部署、启用、扩缩容、许可更新时，优先把“版本差异 + 集群关系 + 节点关系 + 许可资源”一起保留。
- 如果许可不是唯一主轴，也不能丢掉部署形态、节点关系和版本分支。
- 如果需求同时涉及提示/告警和网络编辑，必须同时保留弹窗/表单子树与提示/告警固定块，禁止把提示类内容并入普通字段校验。

## 禁止项

- 禁止把 ER 3.4 及以上与低于 3.4 的差异压成同一组。
- 禁止把普通集群、5.x 双活、6.x 双活等不同部署形态混在同一个字段分组下。
- 禁止把“字段必填 + 格式校验 + 过滤逻辑 + 提交结果”压成一条 case。
- 禁止忽略事件审计、权限管理、异常操作限制这三类运维高频固定块。
- 禁止把 `管理网络 + TEP 网络`、`字段可编辑性 + 提交结果`、`批量填充 + 自动分配`、`提示弹窗 + 信息告警` 压成单条 case。
- 禁止在许可类运维需求里只看许可，不看部署版本、集群关系和节点关系。
- 禁止把样本中的“复合点击”或“批量填充后继续操作”原样落为单个 `[C]`。
