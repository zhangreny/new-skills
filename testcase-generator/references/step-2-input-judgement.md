# 第二步：判断输入并返回确认项

## 目标

基于 `input-manifest.json` 判断本次有哪些可用输入、哪些本地路径真实存在、每个目录下会纳入哪些文件，以及此时应该向用户返回哪一种确认项。

## 执行顺序

1. 如果当前轮存在用户通过 agent 上传的文档，先把这些文档保存到第一步创建的工作目录。
2. 保存后，把 `input-manifest.json` 中 `uploaded files by agent` 更新为这些已保存文件的绝对路径。
3. 运行 `scripts/inspect_input_manifest.py <input-manifest.json 绝对路径>`。
4. 直接阅读脚本标准输出返回的 JSON 结果。
5. 必须先把结果发给用户确认。
6. 如果用户继续补充新的输入，先更新 `input-manifest.json`；如果补充的是 agent 上传文档，先保存并回写绝对路径，再重新执行第二步。
7. 只有在用户明确回复“确认继续”或等效确认后，才能继续第三步。

## 脚本职责

脚本至少要完成以下动作：

- 读取 `input-manifest.json`。
- 解析四类输入数组。
- 遍历 `user file directory` 中的每一项。
- 如果某一项是目录，则递归读取该目录下全部 `.md` 和 `.markdown` 文件。
- 如果某一项是文件，则检查文件是否存在。
- 生成“本次拟使用的文件清单”。
- 生成“建议确认项编号”和“建议确认文案”。
- 直接把结果返回给调用方，不额外写中间结果文件。

上传文档的保存动作不由该脚本负责，而是在第二步进入确认前由 agent 先完成。
`uploaded files by agent` 在第二步中应是工作目录下已保存文件的绝对路径列表。

## 确认项规则

### 确认项 1

触发条件：

- `figma url` 非空。
- `google doc url`、`uploaded files by agent`、`user file directory` 都为空。

返回给用户的确认重点：

- 已收到 Figma 链接。
- 当前没有配套需求资料。
- 询问用户是否需要补充上传该 Figma 对应 feature 的需求文件。

### 确认项 2

触发条件：

- `figma url` 为空。

返回给用户的确认重点：

- 当前没有 Figma 设计稿。
- 询问用户是否需要补充上传该 feature 相关的 Figma 设计稿。

### 确认项 3

触发条件：

- `figma url` 非空。
- 同时至少存在一种其他输入。

返回给用户的确认重点：

- 已同时收到 Figma 和其他资料。
- 展示本次拟使用的链接、目录和文件清单。
- 询问用户是否全部纳入，或指出要排除的文件。

## 输出要求

- 第二步回复必须面向用户可直接阅读，不要只贴脚本原始 JSON。
- 如果当前轮存在用户通过 agent 上传的文档，先保存到第一步创建的工作目录，再把 `uploaded files by agent` 更新为绝对路径，并把保存结果展示给用户确认。
- 当 `user file directory` 是目录时，只展示其中的 Markdown 文件。
- 当路径不存在时，要明确写出“路径不存在”。
- 没有用户明确确认前，不能进入第三步。
- 如果用户在第二步期间又补充了新输入，必须重新执行第二步并再次确认，直到用户明确回复“确认继续”。

## 特殊情况

- 如果四类输入都为空：直接提示没有可处理资料，请用户至少补充一种输入。
- 如果本地路径不存在：在确认结果中明确标记为“路径不存在”，不要假装读取成功。
- 如果目录很大：仍然列出其中的 Markdown 文件，但按相对路径组织，方便用户排除无关文件。
- 如果用户尚未明确确认：停止在第二步，不要默认继续生成测试用例。
