# Step6-7 维度与组合输出模板

默认的维度拆解结果和组合展开结果都使用“树形缩进列表”，不是 markdown `#` 标题。

## 输出目标

- 每一行都必须以 `- ` 开头。
- 通过四个空格表示一层缩进，不使用两个空格、Tab 或其他缩进宽度。
- 层级无限，不限制只能写到几层。
- Step6 和 Step7 的输出都禁止出现 `[C] ` 前缀。
- `描述：` 和 `来源：` 也必须写成带 `- ` 的独立行，并且相对于所属节点多缩进一层。

## 层级规则

- 树形层级按内容自然展开，不预设固定层数。
- 分组节点可以表达业务域、功能模块、页面、对象、状态、场景、条件、校验维度、许可画像、环境前提、组合方式等任意合理层次。
- Step6 重点是“维度拆解”，应先按业务场景分组，再按测试变量继续细分。
- Step7 重点是“组合展开”，应在已确认的维度拆解基础上继续补出可执行的组合层。
- 如果当前节点下面仍然存在明显的子场景、子状态、子对象、子条件或子变量，就应该先继续分组，不要过早收束。
- 同一层内的颗粒度必须一致，不要把“场景分组”和“具体组合结论”混在同一层。

## 最小示意

```md
- 某分组节点
    - 描述：说明该节点覆盖的范围、边界或校验维度。
    - 来源：通过《某文档.md》中的“某内容”得出。
    - 某下层分组
        - 描述：说明该下层分组承接的变量、前提或组合方向。
        - 来源：通过《某文档.md》中的“某规则”内容，结合某截图中的“某区域”内容得出。
```

## 扩展示意

```md
- 某上层分组
    - 描述：说明上层范围。
    - 来源：通过《某文档.md》中的“某上层规则”内容得出。
    - 某中间分组
        - 描述：说明该层表示某类维度、前提或组合方向。
        - 来源：通过《某文档.md》中的“某规则”内容，结合某截图中的“某区域”内容得出。
        - 某下层分组
            - 描述：说明该层进一步展开后的条件、状态或结果方向。
            - 来源：通过《某文档.md》中的“某具体规则”内容，结合某截图中的“某具体界面元素”内容得出。
```

## 硬性约束

- 所有非空行都必须是树形列表项，必须以 `- ` 开头。
- 只允许使用四个空格做层级缩进。
- 每个节点必须紧跟两行子项：`描述：`、`来源：`。
- `描述：` 和 `来源：` 的缩进必须比所属节点多四个空格。
- 一个节点写完 `描述：` 和 `来源：` 后，才可以继续写它的子节点。
- Step6 和 Step7 输出中禁止出现 `[C] ` 前缀。
- 不写编号，例如 `1.`、`TC001`、`case-01`。
- 不写表格，不写游离段落，不写“同上”“略”“待补充”。

## 描述写法

- 分组节点下的 `描述：` 用来说明这一组内容承接的业务范围、测试变量、环境前提或组合方向，不写执行步骤。
- `描述：` 要能让人看出这一层为什么存在，以及后续为什么还需要继续展开。
- `描述：` 不要只写“正常流程”“异常场景”“边界情况”这种过空的话，必须落到对象、前提、变量或预期类型。

## 来源写法

- `来源：` 不能只写文件名。
- `来源：` 必须写成“通过 xx 文件的 xx 内容得出”的形式，尽量指出具体依据。
- 引用需求文档时，优先写成：`通过《xxx.md》中的“某条规则/某字段定义/某段流程/某个限制条件”内容得出。`
- 引用截图时，优先写成：`通过 xxx.png 中的“某页面区域/某控件/某弹窗/某报错文案/某展示字段”内容得出。`
- 同时使用多份依据时，写成：`通过《xxx.md》中的“……”内容，结合 xxx.png 中的“……”内容得出。`
- 禁止只写：`来源：需求文档`、`来源：设计稿`、`来源：LB-license-CPU-socket.md` 这种过粗来源。
