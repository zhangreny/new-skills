# Step6 产品理解输出模板

Step6 的目标不是直接开始拆测试维度，而是先把“有助于理解当前材料的产品资料”筛出来，并说明这些资料如何帮助理解需求文档和图片。

## 固定结构

输出文件 `step6_product_context.md` 必须包含以下二级标题，且顺序固定：

```md
# Step6 产品理解
## 已扫描的产品资料
## 选中的产品资料
## 未采用的产品资料
## 产品对象与术语校准
## 黄金参考候选与结论
## 测试资源类型
## 资源前提与消耗/归还规则
## 对需求文档和图片理解的帮助
## 版本边界与提醒
```

## 写法要求

- `已扫描的产品资料`：列出本次检查过的 SmartX docs 文件，至少写文件名和类别。
- `选中的产品资料`：逐条写“选中了什么、为什么有助于理解当前材料”，不能只写文件名。
- `未采用的产品资料`：逐条写“为什么当前暂不需要”，避免无声忽略。
- `产品对象与术语校准`：提炼对后续蓝图有帮助的产品对象、对象关系、名词解释。
- `黄金参考候选与结论`：明确写“已采用黄金参考”或“未采用黄金参考”，并说明原因。若已采用，写清参考路径和采用理由。
- `测试资源类型`：按通用资源类型和产品特有资源类型，归纳当前任务实际涉及的资源。
- `资源前提与消耗/归还规则`：写清哪些资源是前置资源，哪些资源会消耗、占用、归还或决定结果分支。
- `对需求文档和图片理解的帮助`：明确说明这些资料帮助识别了哪些页面、字段组、对象层级、前置条件或能力边界。
- `版本边界与提醒`：如果材料涉及版本差异、启用条件、部署前提或兼容限制，必须显式写出。
- Step6 只能做“理解增强”和“术语校准”，不得在这里直接输出 testcase、组合树或最终结论。

## 好的内容方向

- 管理指南可用于确认页面入口、对象层级、前置条件、字段组和配置步骤。
- 技术白皮书可用于确认对象关系、网络模式、主备关系、能力边界和关键术语。
- 术语表可用于统一需求文档、设计稿和历史样本中的命名。
- 发布说明可用于解释版本分支为何必须保留。
- 许可类需求中，应提前识别许可对象、许可方式、许可类型、额度状态、过期状态和消耗/归还动作。

## 禁止项

- 禁止因为 SmartX docs 中“提到过”某功能，就把它强行加入当前需求范围。
- 禁止把白皮书内容直接抄成测试用例。
- 禁止跳过“未采用的产品资料”这一节。
- 禁止默认写死某一个黄金参考。
