# Step6 用例输出模板

默认的 UI 前端测试用例输出结构为“树形缩进列表”，不是 markdown `#` 标题。

## 输出目标

- 每一行都必须以 `- ` 开头。
- 通过四个空格表示一层缩进，不使用两个空格、Tab 或其他缩进宽度。
- 层级无限，不限制只能写到几层。
- 具体测试用例必须带 `[C] ` 前缀。
- 分组节点禁止带 `[C] ` 前缀。
- `描述：` 和 `来源：` 也必须写成带 `- ` 的独立行，并且相对于所属节点多缩进一层。

## 层级规则

- 树形层级按内容自然展开，不预设固定层数。
- 分组节点可以表达业务域、功能模块、页面、对象、状态、场景、条件、校验维度等任意合理层次。
- `[C]` 具体测试用例应挂在“当前最细、但仍有分组意义”的父节点下，不要求固定挂在第几层。
- 如果当前节点下面已经不需要再分组，就直接写 `[C]` 用例，不要为了凑层级额外新增空洞分组。
- 如果当前节点下面仍然存在明显的子场景、子状态、子对象或子条件，就应该先继续分组，再落到 `[C]` 用例。

## 最小示意

下面示例仅用于说明树形结构、`描述：`、`来源：` 和 `[C]` 的位置关系，不代表固定层数，也不代表必须使用“业务域 -> 功能模块 -> 场景分类”的套路。

```md
- 某分组节点
    - 描述：说明该节点覆盖的范围、边界或校验维度。
    - 来源：通过《某文档.md》中的“某内容”得出。
    - [C] 某具体测试用例
        - 描述：验证某对象在某条件下执行某操作时，结果符合预期。
        - 来源：通过《某文档.md》中的“某规则/某字段定义/某限制条件”内容，结合某截图中的“某区域/某控件/某提示”内容得出。
```

## 扩展示意

如果某个分组下面仍需继续细分，可以继续增加中间层，层数不限。

```md
- 某上层分组
    - 描述：说明上层范围。
    - 来源：通过《某文档.md》中的“某上层规则”内容得出。
    - 某下层分组
        - 描述：说明下层范围。
        - 来源：通过《某文档.md》中的“某下层规则”内容，结合某截图中的“某区域”内容得出。
        - [C] 某具体测试用例
            - 描述：验证某功能在某条件下的结果符合预期。
            - 来源：通过《某文档.md》中的“某具体规则”内容，结合某截图中的“某具体界面元素”内容得出。
```

## 硬性约束

- 所有非空行都必须是树形列表项，必须以 `- ` 开头。
- 只允许使用四个空格做层级缩进。
- 每个节点必须紧跟两行子项：`描述：`、`来源：`。
- `描述：` 和 `来源：` 的缩进必须比所属节点多四个空格。
- 一个节点写完 `描述：` 和 `来源：` 后，才可以继续写它的子节点。
- 具体测试用例节点必须以 `[C] ` 开头。
- 非具体测试用例节点禁止以 `[C] ` 开头。
- 不写编号，例如 `1.`、`TC001`、`case-01`。
- 不写表格，不写游离段落，不写“同上”“略”“待补充”。
- 同一层内的颗粒度必须一致，不要把“功能模块”和“具体测试用例”混在同一层。

## 描述写法

- 分组节点下的 `描述：` 用来说明这一组内容的覆盖范围、业务边界或校验维度，不写执行步骤。
- 具体测试用例下的 `描述：` 用来说明这条用例到底验证什么，建议写成“验证……在……条件下……结果符合预期”的单句。
- `描述：` 要能让人不看父节点也大致知道这一条在测什么。
- `描述：` 不要只写“正常流程”“异常场景”“校验展示”这种过空的话，必须落到对象、条件和预期结果。

## 来源写法

- `来源：` 不能只写文件名。
- `来源：` 必须写成“通过 xx 文件的 xx 内容得出”的形式，尽量指出具体依据。
- 引用需求文档时，优先写成：`通过《xxx.md》中的“某条规则/某字段定义/某段流程/某个限制条件”内容得出。`
- 引用截图时，优先写成：`通过 xxx.png 中的“某页面区域/某控件/某弹窗/某报错文案/某展示字段”内容得出。`
- 同时使用多份依据时，写成：`通过《xxx.md》中的“……”内容，结合 xxx.png 中的“……”内容得出。`
- 禁止只写：`来源：需求文档`、`来源：设计稿`、`来源：LB-license-CPU-socket.md` 这种过粗来源。

## 好坏示例

坏示例：

```md
- [C] 创建成功
    - 描述：正常流程
    - 来源：LB-license-CPU-socket.md、设计稿
```

好示例：

```md
- [C] 创建首个负载均衡实例时按所属集群 CPU 插槽总数扣减许可
    - 描述：验证在目标集群首次创建负载均衡实例时，系统按该集群全部主机的 CPU 插槽总数扣减许可，且创建成功后许可卡片中的已用额度同步增加。
    - 来源：通过《LB-license-CPU-socket.md》中的“每在 SMTX 集群中部署任一 LB 实例，消耗的许可数等于集群内所有主机的物理 CPU 插槽总数”内容，结合 02_LBI_11-4014.png 中的“创建 LBI/部署 ER 服务相关界面”内容得出。
```
