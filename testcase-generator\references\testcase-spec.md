# 测试用例输出规范

## 产物

最少生成以下文件：

1. `requirements-summary.md`
2. `testcases.md`
3. `coverage-matrix.md`
4. `open-questions.md`

## requirements-summary.md

按下面结构组织：

```markdown
# 需求总结

## 已确认信息

## 角色与权限

## 关键业务规则

## 页面与流程

## 假设与缺口
```

## testcases.md

优先使用“每条用例一个二级标题”的形式，避免长表格难以维护。

推荐模板：

```markdown
# 测试用例

## TC-001 登录-正确账号密码登录成功

- 模块：登录
- 优先级：P0
- 类型：正向
- 来源文件：source-google-doc.md, source-figma.md
- 前置条件：
  - 用户已注册
- 测试数据：
  - 账号：valid_user
  - 密码：ValidPassword123
- 测试步骤：
  1. 打开登录页。
  2. 输入正确账号和密码。
  3. 点击“登录”按钮。
- 预期结果：
  1. 登录成功。
  2. 跳转到首页。
  3. 顶部展示用户昵称。
```

## coverage-matrix.md

至少覆盖功能点、来源文件和用例编号三列。

推荐模板：

```markdown
# 覆盖矩阵

| 功能点 | 来源文件 | 用例编号 |
| --- | --- | --- |
| 登录成功 | source-google-doc.md, source-figma.md | TC-001 |
```

## open-questions.md

把以下内容单独列出来：

- 来源之间冲突的信息
- 无法确认的字段规则
- 未明确的异常处理
- 依赖用户或业务方确认的策略

## 覆盖要求

每个关键功能点默认覆盖以下维度：

- 至少 1 条正向主流程
- 至少 1 条异常或失败流程
- 至少 1 条边界或空值场景
- 涉及权限时至少 1 条角色差异场景
- 涉及状态变更时至少 1 条重复操作或逆序操作场景

如果 Figma 明确给出以下信息，也要纳入测试：

- 字段必填和输入限制
- 按钮可点击态与禁用态
- 错误提示、空态、成功态
- 弹窗、抽屉、Toast、跳转和返回行为

## 优先级建议

- `P0`：核心交易链路、登录鉴权、支付、提交流程、关键权限控制
- `P1`：主路径上的重要分支、关键校验、常见异常
- `P2`：次要分支、易用性提示、一般边界场景
- `P3`：低风险补充验证、非关键文案或表现细节

## 写作原则

- 只写可执行步骤，不写含糊描述。
- 预期结果要能被验证，不要只写“功能正常”。
- 未确认信息单列，不要混进确定性用例中。
- 尽量为每条用例标注来源文件，便于回溯。
